href="/wiki/Query_language" title="Query language">query</a> and <a href="/wiki/Functional_programming" title="Functional programming">functional programming</a> language that queries and transforms collections of structured and <a href="/wiki/Unstructured_data" title="Unstructured data">unstructured data</a>, usually in the form of <a href="/wiki/XML" title="XML">XML</a>, text and with vendor-specific extensions for other data formats (<a href="/wiki/JSON" title="JSON">JSON</a>, <a href="/wiki/Binary_data" title="Binary data">binary</a>, etc.). The language is developed by the XML Query <a href="/wiki/Working_group" title="Working group">working group</a> of the <a href="/wiki/W3C" class="mw-redirect" title="W3C">W3C</a>. The work is closely coordinated with the development of <a href="/wiki/XSLT" title="XSLT">XSLT</a>  by the XSL Working Group; the two groups share responsibility for <a href="/wiki/XPath" title="XPath">XPath</a>, which is a subset of XQuery.\n</p><p><b>XQuery 1.0</b> became a <a href="/wiki/W3C_Recommendation" class="mw-redirect" title="W3C Recommendation">W3C Recommendation</a> on January 23, 2007.<sup id="cite_ref-4" class="reference"><a href="#cite_note-4">&#91;4&#93;</a></sup>\n</p><p><b>XQuery 3.0</b> became a <a href="/wiki/W3C_Recommendation" class="mw-redirect" title="W3C Recommendation">W3C Recommendation</a> on April 8, 2014.<sup id="cite_ref-5" class="reference"><a href="#cite_note-5">&#91;5&#93;</a></sup>\n</p><p><b>XQuery 3.1</b> became a <a href="/wiki/W3C_Recommendation" class="mw-redirect" title="W3C Recommendation">W3C Recommendation</a> on March 21, 2017.<sup id="cite_ref-6" class="reference"><a href="#cite_note-6">&#91;6&#93;</a></sup>\n</p>\n<style data-mw-deduplicate="TemplateStyles:r996844942">.mw-parser-output .templatequote{overflow:hidden;margin:1em 0;padding:0 40px}.mw-parser-output .templatequote .templatequotecite{line-height:1.5em;text-align:left;padding-left:1.6em;margin-top:0}</style><blockquote class="templatequote"><p>"The mission of the XML Query project is to provide flexible query facilities to extract data from real and virtual documents on the World Wide Web, therefore finally providing the needed interaction between the Web world and the database world. class="toclevel-1 tocsection-16"><a href="#External_links"><span class="tocnumber">10</span> <span class="toctext">External links</span></a></li>\n</ul>\n</div>\n\n<h2><span class="mw-headline" id="Features">Features</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=XQuery&amp;action=edit&amp;section=1" title="Edit section: Features">edit</a><span class="mw-editsection-bracket">]</span></span></h2>\n<p>XQuery is a functional, <a href="/wiki/Side_effect_(computer_science)" title="Side effect (computer science)">side effect-free</a>, expression-oriented programming language with a simple <a href="/wiki/Type_system" title="Type system">type system</a>, summed up by Kilpel\xc3\xa4inen:<sup id="cite_ref-8" class="reference"><a href="#cite_note-8">&#91;8&#93;</a></sup>\n</p>\n<link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r996844942"/><blockquote class="templatequote"><p>All XQuery expressions operate on sequences, and evaluate to sequences. <i>Sequences</i> are ordered lists of items. <i>Items</i> can be either <i>nodes</i>, which represent components of XML documents, or <i>atomic values</i>, which are instances of XML Schema base types like <style data-mw-deduplicate="TemplateStyles:r886049734">.mw-parser-output .monospaced{font-family:monospace,monospace}</style><span class="monospaced">xs:integer</span> or <link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r886049734"/><span class="monospaced">xs:string</span>. Sequences can also be empty, or consist of a single item only. No distinction is made between a single item and a singleton sequence. (...) XQuery/XPath sequences differ from lists in languages like <a href="/wiki/Lisp_(programming_language)" title="Lisp (programming language)">Lisp</a> and <a href="/wiki/Prolog" title="Prolog">Prolog</a> by excluding nested sequences. Designers of XQuery may have considered nested sequences unnecessary for the manipulation of document contents. Nesting, or hierarchy of document structures is instead represented by nodes and their child-parent relationships</p></blockquote>\n<p>XQuery provides the means to extract and manipulate data from XML documents or any data source that can be viewed as XML, such as <a href="/wiki/Relational_database" title="Relational database">relational databases</a><sup id="cite_ref-9" class="reference"><a href="#cite_note-9">&#91;9&#93;</a></sup> or office documents.\n</p><p>XQuery contains a superset of <a href="/wiki/XPath" title="XPath">XPath</a> expression syntax to address specific parts of an XML document. It supplements this with a <a href="/wiki/SQL" title="SQL">SQL</a>-like "<a href="/wiki/FLWOR" title="FLWOR">FLWOR</a> expression" for performing joins. A FLWOR expression is constructed from the five clauses after which it is named: FOR, LET, WHERE, ORDER BY, RETURN.\n</p><p>The language also provides syntax allowing new XML documents to be constructed. Where the element and attribute names are known in advance, an XML-like syntax can be used; in other cases, expressions referred to as dynamic node constructors are available. All these constructs are defined as expressions within the language, and can be arbitrarily nested.\n</p><p>The language is based on the <a href="/wiki/XQuery_and_XPath_Data_Model" title="XQuery and XPath Data Model">XQuery and XPath Data Model</a> (XDM) which uses a tree-structured model of the information content of an XML document, containing seven kinds of nodes: document nodes, elements, attributes, text nodes, comments, processing instructions, and namespaces.\n</p><p>XDM also models all values as sequences (a singleton value is considered to be a sequence of length one). The items in a sequence can either be XML nodes or atomic values. Atomic values may be integers, strings, booleans, and so on: the full list of types is based on the primitive types defined in <a href="/wiki/XML_Schema_(W3C)" title="XML Schema (W3C)">XML Schema</a>.\n</p><p>Features for updating XML documents or databases, and <a href="/wiki/Full_text_search" class="mw-redirect" title="Full text search">full text search</a> capability, are not part of the core language, but are defined in add-on extension standards: XQuery Update Facility 1.0 supports update feature and XQuery and XPath Full Text 1.0 supports full text search in XML called).\n</p>\n<h2><span class="mw-headline" id="Examples">Examples</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a As such, its capabilities overlap with <a href="/wiki/XSLT" title="XSLT">XSLT</a>, which was designed expressly to allow input XML documents to be transformed into HTML or other formats.\n</p><p>The XSLT 2.0 and XQuery standards were developed by separate working groups within <a href="/wiki/W3C" class="mw-redirect" title="W3C">W3C</a>, working together to ensure a common approach where appropriate. They share the same data model (<a href="/wiki/XQuery_and_XPath_Data_Model" title="XQuery and XPath Data Model">XDM</a>), type system, and function library, and both include <a href="/wiki/XPath" title="XPath">XPath</a> 2.0 as a sublanguage.\n</p>\n<h3><span class="mw-headline" id="Origin">Origin</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=XQuery&amp;action=edit&amp;section=6" title="Edit section: Origin">edit</a><span class="mw-editsection-bracket">]</span></span></h3>\n<p>The two languages, however, are rooted in different traditions and serve the needs of different communities. XSLT was primarily conceived as a stylesheet language whose primary goal was to render XML for the human reader on screen, on <a href="/wiki/The_web" class="mw-redirect" title="The web">the web</a> (as <a href="/wiki/Web_template" class="mw-redirect" title="Web template">web template language</a>), or on paper. XQuery was primarily conceived as a <a href="/wiki/Database_query_language" class="mw-redirect" title="Database query language">database query language</a> in the tradition of <a href="/wiki/SQL" title="SQL">SQL</a>.\n</p><p>Because the two languages originate in different communities, XSLT is stronger<sup class="noprint Inline-Template" style="margin-left:0.1em; white-space:nowrap;">&#91;<i><a href="/wiki/Wikipedia:Manual_of_Style/Words_to_watch#Unsupported_attributions" title="Wikipedia:Manual of Style/Words to watch"><span title="The material near this tag may use weasel words or too-vague attribution. (May 2015)">according to whom?</span></a></i>&#93;</sup> in its handling of narrative documents with more flexible structure, while XQuery is stronger in its data handling (for example, when performing relational joins).\n</p>\n<h3><span class="mw-headline" id="Versions">Versions</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=XQuery&amp;action=edit&amp;section=7" title="Edit section: Versions">edit</a><span class="mw-editsection-bracket">]</span></span></h3>\n<p>XSLT 1.0 appeared as a Recommendation in 1999, whereas XQuery 1.0 only became a Recommendation in early 2007; as a result, XSLT is still much more widely used. Both languages have similar expressive power, though XSLT 2.0 has many features that are missing from XQuery 1.0, such as grouping, number and date formatting, and greater control over XML namespaces.<sup id="cite_ref-Kay_10-0" class="reference"><a href="#cite_note-Kay-10">&#91;10&#93;</a></sup><sup id="cite_ref-Eisenberg_11-0" class="reference"><a href="#cite_note-Eisenberg-11">&#91;11&#93;</a></sup><sup id="cite_ref-Smith_12-0" class="reference"><a href="#cite_note-Smith-12">&#91;12&#93;</a></sup>  Many of these features were planned for XQuery 3.0.<sup id="cite_ref-XQuery11_13-0" class="reference"><a href="#cite_note-XQuery11-13">&#91;13&#93;</a></sup>\n</p><p>Any comparison must take into account the fact that XSLT 1.0 and XSLT 2.0 are very different languages. XSLT 2.0, in particular, has been heavily influenced by XQuery in its move to strong typing and schema-awareness.\n</p>\n<h3><span class="mw-headline" id="Strengths_and_weaknesses">Strengths and weaknesses</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=XQuery&amp;action=edit&amp;section=8" title="Edit section: Strengths and weaknesses">edit</a><span class="mw-editsection-bracket">]</span></span></h3>\n<p>Usability studies have shown that XQuery is easier to learn than <a href="/wiki/XSLT" title="XSLT">XSLT</a>, especially for users with previous experience of database languages such as SQL.<sup id="cite_ref-14" class="reference"><a href="#cite_note-14">&#91;14&#93;</a></sup> This can be attributed to the fact that XQuery is a smaller language with fewer concepts to learn, and to the fact that programs are more concise. It is also true that XQuery is more orthogonal, in that any expression can be used in any syntactic context. By contrast, XSLT is a two-language system in which XPath expressions can be nested in XSLT instructions but not vice versa.\n</p><p>XSLT is currently stronger than XQuery for applications that involve making small changes to\na document (for example, deleting all the NOTE elements). Such applications are generally handled\nin XSLT by use of a coding pattern that involves an identity template that copies all nodes unchanged, modified by specific templates that modify selected nodes. XQuery has no equivalent to this coding pattern, though in future versions it will be possible to tackle such problems using the update facilities in the language that are under development.<sup id="cite_ref-XQueryUpdate_15-0" class="reference"><a href="#cite_note-XQueryUpdate-15">&#91;15&#93;</a></sup>\n</p><p>XQuery 1.0 lacked any kind of mechanism for dynamic binding or polymorphism; this has been remedied with the introduction of functions as first-class values in XQuery 3.0. The absence of this capability starts to become noticeable when writing large applications, or when writing code that is designed to be reusable in different environments.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">&#91;<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (May 2015)">citation needed</span></a></i>&#93;</sup> XSLT offers two complementary mechanisms in this area: the dynamic matching of template rules, and the ability to override rules using <code>xsl:import</code>, that make it possible to write applications with multiple customization layers.\n</p><p>The absence of these facilities from XQuery 1.0 was a deliberate design decision: it has the consequence that XQuery is very amenable to static analysis, which is essential to achieve the level of optimization needed in database query languages. This also makes it easier to detect errors in XQuery code at compile time.\n</p><p>The fact that XSLT 2.0 uses <a href="/wiki/XML" title="XML">XML</a> syntax makes it rather verbose in comparison to XQuery 1.0. However, many large applications take advantage of this capability by using XSLT to read, write, or modify stylesheets dynamically as part of a processing pipeline. The use of XML syntax also enables the use of XML-based tools for managing XSLT code. By contrast, XQuery syntax is more suitable for embedding in traditional programming languages such as <a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a> (see <a href="/wiki/XQuery_API_for_Java" title="XQuery API for Java">XQuery API for Java</a>) or <a href="/wiki/C_Sharp_(programming_language)" title="C Sharp (programming language)">C#</a>. If necessary, XQuery code can also be expressed in an XML syntax called XQueryX. The XQueryX representation of XQuery code is rather verbose and not convenient for humans, but can easily be processed with XML tools, for example transformed with XSLT stylesheets.<sup id="cite_ref-XQueryX_16-0" class="reference"><a href="#cite_note-XQueryX-16">&#91;16&#93;</a></sup><sup id="cite_ref-17" class="reference"><a href="#cite_note-17">&#91;17&#93;</a></sup>\n</p>\n<h2><span class="mw-headline" id="Extensions_and_future_work">Extensions and future work</span><span class="mw-editsection"><span
