class="mw-parser-output"><div class="shortdescription nomobile noexcerpt noprint searchaux" style="display:none">Service offered by an electronic device to another electronic device, communicating with each other via the World Wide Web</div>\n<p>The term <b>Web service</b> (<b>WS</b>) is either:\n</p>\n<ul><li>a service offered by an electronic device to another electronic device, communicating with each other via the <a href="/wiki/World_Wide_Web" title="World Wide Web">World Wide Web</a>, or</li>\n<li>a server running on a computer device, listening for requests at a particular port over a network, serving web documents (HTML, <a href="/wiki/JSON" title="JSON">JSON</a>, <a href="/wiki/XML" title="XML">XML</a>, images).</li></ul>\n<p>In a Web service a Web technology such as <a href="/wiki/Hypertext_Transfer_Protocol" title="Hypertext Transfer Protocol">HTTP</a> is used for transferring machine-readable file formats such as <a href="/wiki/XMLHttpRequest" title="XMLHttpRequest">XML</a> and <a href="/wiki/JSON" title="JSON">JSON</a>.\n</p><p>In practice, a web service commonly provides an <a href="/wiki/Object_database" title="Object database">object-oriented</a> Web-based interface to a database server, utilized for example by another Web server, or by a <a href="/wiki/Mobile_app_development" title="Mobile app development">mobile app</a>, that provides a user interface to the end-user. Many organizations that provide data in formatted HTML pages will also provide that data on their server as XML or JSON, often through a Web service to allow <a href="/wiki/Web_syndication" title="Web syndication">syndication</a>, for example, Wikipedia\'s <a href="/wiki/Help:Export" title="Help:Export">Export</a>. Another application offered to the end-user may be a <a href="/wiki/Mashup_(web_application_hybrid)" title="Mashup (web application hybrid)">mashup</a>, where a Web server consumes several Web services at different machines and compiles the content into one user interface.\n</p>\n<div id="toc" class="toc" role="navigation" aria-labelledby="mw-toc-heading"><input type="checkbox" (programming)">Ajax (programming)</a></div>\n<p>Asynchronous JavaScript And XML (AJAX) is a dominant technology for Web services. Developing from the combination of HTTP servers, JavaScript clients and <a href="/wiki/Plain_Old_XML" title="Plain Old XML">Plain Old XML</a> (as distinct from SOAP and W3C Web Services), now it is frequently used with <a href="/wiki/JSON_Web_Token" title="JSON Web Token">JSON</a> as well as, or instead of, XML.\n</p>\n<h3><span class="mw-headline" id="REST">REST</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Web_service&amp;action=edit&amp;section=3" title="Edit section: REST">edit</a><span class="mw-editsection-bracket">]</span></span></h3>\n<div role="note" class="hatnote navigation-not-searchable">Main article: <a href="/wiki/REST" class="mw-redirect" title="REST">REST</a></div>\n<p>Representational State Transfer (REST) is an architecture for well-behaved Web services that can function at Internet scale.\n</p><p>In a 2004 document, the W3C sets following REST as a key distinguishing feature of Web services:\n</p>\n<style data-mw-deduplicate="TemplateStyles:r996844942">.mw-parser-output .templatequote{overflow:hidden;margin:1em 0;padding:0 40px}.mw-parser-output .templatequote .templatequotecite{line-height:1.5em;text-align:left;padding-left:1.6em;margin-top:0}</style><blockquote class="templatequote"><p>We can identify two major classes of Web services:\n</p><ul><li><a href="/wiki/REST" class="mw-redirect" title="REST">REST</a>-compliant Web services, in which the primary purpose of the service is to manipulate XML representations of <a href="/wiki/Web_resource" title="Web resource">Web resource</a>s using a uniform set of <a href="/wiki/Stateless_protocol" title="Stateless protocol">stateless</a> operations; and</li>\n<li>arbitrary Web services, in which the service may expose an arbitrary set of operations.<div class="templatequotecite">\xe2\x80\x94&#8201;<cite>W3C, Web Services Architecture<sup id="cite_ref-1" class="reference"><a href="#cite_note-1">&#91;1&#93;</a></sup></cite></div></li></ul></blockquote>\n<h3><span class="mw-headline" id="Web_services_that_use_markup_languages">Web services that use markup languages</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Web_service&amp;action=edit&amp;section=4" title="Edit section: Web services that use markup languages">edit</a><span class="mw-editsection-bracket">]</span></span></h3>\n<p>There are a number of Web services that use markup languages:\n</p>\n<ul><li><a href="/wiki/JSON-RPC" title="JSON-RPC">JSON-RPC</a>.</li>\n<li><a href="/wiki/JSON-WSP" title="JSON-WSP">JSON-WSP</a></li>\n<li><a href="/wiki/Representational_state_transfer" title="Representational title="Web Services Description Language">Web Services Description Language</a> (WSDL), developed by the <a href="/wiki/World_Wide_Web_Consortium" title="World Wide Web Consortium">W3C</a></li>\n<li><a href="/wiki/Web_Services_Flow_Language" title="Web Services Flow Language">Web Services Flow Language</a> (WSFL), superseded by <a href="/wiki/Business_Process_Execution_Language" title="Business Process Execution Language">BPEL</a></li>\n<li><a href="/wiki/Web_template" class="mw-redirect" title="Web template">Web template</a></li>\n<li><a href="/wiki/WS-MetadataExchange" title="WS-MetadataExchange">WS-MetadataExchange</a></li>\n<li><a href="/wiki/XML_Interface_for_Network_Services" title="XML Interface for Network Services">XML Interface for Network Services</a> (XINS), provides a <a href="/wiki/Plain_Old_XML" title="Plain Old XML">POX</a>-style web service specification format</li></ul>\n<h3><span class="mw-headline" id="Web_API">Web API</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Web_service&amp;action=edit&amp;section=5" title="Edit section: Web API">edit</a><span class="mw-editsection-bracket">]</span></span></h3>\n<div role="note" class="hatnote navigation-not-searchable">Main article: <a href="/wiki/Web_API" title="Web API">Web API</a></div>\n<p>A <a href="/wiki/Web_API" title="Web API">Web API</a> is a development in Web services where emphasis has been moving to simpler <a href="/wiki/Representational_state_transfer" title="Representational state transfer">representational state transfer</a> (REST) based communications.<sup id="cite_ref-2" class="reference"><a href="#cite_note-2">&#91;2&#93;</a></sup> Restful APIs do not require XML-based Web service protocols (<a href="/wiki/SOAP" title="SOAP">SOAP</a> and WSDL) to support their interfaces.\n</p>\n<h2><span class="mw-headline" id="W3C_Web_services">W3C Web services</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Web_service&amp;action=edit&amp;section=6" title="Edit section: W3C Web services">edit</a><span class="mw-editsection-bracket">]</span></span></h2>\n<p>In relation to W3C Web services, the <a href="/wiki/W3C" class="mw-redirect" title="W3C">W3C</a> defined a Web service as:\n</p>\n<link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r996844942"/><blockquote class="templatequote"><p>A web service is a software system designed to support interoperable machine-to-machine interaction over a network. It has an interface described in a machine-processable format (specifically <a href="/wiki/Web_Services_Description_Language" title="Web Services Description Language">WSDL</a>). Other systems interact with the web service in a manner prescribed by its description using <a href="/wiki/SOAP" title="SOAP">SOAP</a>-messages, typically conveyed using <a href="/wiki/HTTP" class="mw-redirect" title="HTTP">HTTP</a> with an <a href="/wiki/XML" title="XML">XML</a> <a href="/wiki/Serialization" title="Serialization">serialization</a> in conjunction with other web-related standards.</p><div class="templatequotecite">\xe2\x80\x94&#8201;<cite>W3C, Web Services Glossary<sup id="cite_ref-W3WSG_3-0" class="reference"><a href="#cite_note-W3WSG-3">&#91;3&#93;</a></sup></cite></div></blockquote>\n<p>W3C Web Services may use <a href="/wiki/SOAP" title="SOAP">SOAP</a> over HTTP protocol, allowing less costly (more efficient) interactions over the Internet than via proprietary solutions like EDI/B2B. Besides SOAP over HTTP, Web services can also be implemented on other reliable transport mechanisms like <a href="/wiki/File_Transfer_Protocol" title="File Transfer Protocol">FTP</a>. In a 2002 document, the <a rel="nofollow" class="external text" href="https://www.w3.org/2002/ws/arch/">Web Services Architecture Working Group</a> defined a Web services architecture, requiring a standardized implementation of a "Web service."\n</p>\n<h3><span class="mw-headline" id="Explanation">Explanation</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Web_service&amp;action=edit&amp;section=7" title="Edit section: Explanation">edit</a><span class="mw-editsection-bracket">]</span></span></h3>\n<div class="thumb tright"><div class="thumbinner" style="width:222px;"><a href="/wiki/File:Webservices-en.svg" class="image"><img alt="" src="//upload.wikimedia.org/wikipedia/commons/thumb/5/5e/Webservices-en.svg/220px-Webservices-en.svg.png" decoding="async" width="220" height="186" class="thumbimage" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/5/5e/Webservices-en.svg/330px-Webservices-en.svg.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/5/5e/Webservices-en.svg/440px-Webservices-en.svg.png 2x" data-file-width="716" data-file-height="606" /></a>  <div class="thumbcaption"><div class="magnify"><a href="/wiki/File:Webservices-en.svg" class="internal" title="Enlarge"></a></div>Web services architecture: the service provider sends a WSDL file to UDDI. The service requester contacts UDDI to find out who is the provider for the data it needs, and then it contacts the service provider using the SOAP protocol. The service provider validates the service request and sends structured data in an XML file, using the SOAP protocol. This XML file would be validated again by the service requester using an XSD file.</div></div></div>\n<p>The term "Web service" describes a standardized way of integrating Web-based applications using the <a href="/wiki/XML" title="XML">XML</a>, SOAP, WSDL and UDDI open standards over an <a href="/wiki/Internet_Protocol" title="Internet Protocol">Internet Protocol</a> backbone. XML is the data format used to contain the data and provide metadata around it, SOAP is used to transfer the data, WSDL is used for describing the services available and UDDI lists what services are available.\n</p><p>A Web service is a method of communication between two electronic devices over a network. It is a software function provided at a <a href="/wiki/Network_address" title="Network address">network address</a> over the Web with the service <i>always-on</i> as in the concept of utility computing.\n</p><p>Many organizations use multiple software systems for management.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">&#91;<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (August 2015)">citation needed</span></a></i>&#93;</sup> Different software systems often need to exchange data with each other, and a Web service is a method of communication that allows two software systems to exchange this data over the Internet. The software system that requests data is called a <i>service requester</i>, whereas the software system that would process the request and provide the data is called a <i><a href="/wiki/Service_provider" title="Service provider">service provider</a></i>.\n</p><p>Different software may use different programming languages, and hence there is a need for a method of <a href="/wiki/Data_exchange" title="Data exchange">data exchange</a> that doesn\'t depend upon a particular programming language. Most types of software can, however, interpret XML tags. Thus, Web services can use XML files for data exchange.\n</p><p>Rules for communication different systems need to be defined, such as:\n</p>\n<ul><li>How one system can request data from another system.</li>\n<li>Which specific parameters are needed in the data request.</li>\n<li>What would be the structure of the data produced. (Normally, data is exchanged in XML files, and the structure of the XML file is validated against a .xsd file.)</li>\n<li>What error messages to display when a certain rule for communication is not observed, to make <a href="/wiki/Troubleshooting" title="Troubleshooting">troubleshooting</a> easier.</li></ul>\n<p>All of these rules for communication are defined in a file called <a href="/wiki/Web_Services_Description_Language" title="Web Services Description Language">WSDL</a> (Web Services Description Language), which has a <code>.wsdl</code> extension. (Proposals for <b>Autonomous Web Services</b> (<b>AWS</b>) seek to develop more flexible Web services that do not rely on strict rules.<sup id="cite_ref-5" class="reference"><a href="#cite_note-5">&#91;a&#93;</a></sup>)\n</p><p>A directory called <a href="/wiki/Universal_Description_Discovery_and_Integration" class="mw-redirect" title="Universal Description Discovery and Integration">UDDI</a> (Universal Description, Discovery, and Integration) defines which software system should be contacted for which type of data. So when one software system needs one particular report/data, it would go to the UDDI and find out which other systems it can contact for receiving that data. Once the software system finds out which other systems it should contact, it would then contact that system using a special protocol called <a href="/wiki/SOAP" title="SOAP">SOAP</a> (Simple Object Access Protocol). The service provider system would first validate the data request by referring to the WSDL file, and then process the request and send the data under the SOAP protocol.\n</p>\n<h3><span class="mw-headline" id="Automated_design_methods">Automated design methods<span class="anchor" id="Design_methods"></span></span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Web_service&amp;action=edit&amp;section=8" title="Edit section: Automated design methods">edit</a><span class="mw-editsection-bracket">]</span></span></h3>\n<div class="thumb tright"><div class="thumbinner" style="width:442px;"><a href="/wiki/File:SOA_Detailed_Diagram.png" class="image"><img alt="" src="//upload.wikimedia.org/wikipedia/commons/thumb/8/84/SOA_Detailed_Diagram.png/440px-SOA_Detailed_Diagram.png" decoding="async" width="440" height="232" class="thumbimage" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/8/84/SOA_Detailed_Diagram.png/660px-SOA_Detailed_Diagram.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/8/84/SOA_Detailed_Diagram.png/880px-SOA_Detailed_Diagram.png 2x" data-file-width="1604" data-file-height="845" /></a>  <div class="thumbcaption"><div class="magnify"><a href="/wiki/File:SOA_Detailed_Diagram.png" class="internal" title="Enlarge"></a></div>Web services in a <a href="/wiki/Service-oriented_architecture" title="Service-oriented architecture">service-oriented architecture</a>.</div></div></div>\n<p>Automated tools can aid in the creation of a Web service. For services using WSDL, it is possible to either automatically generate WSDL for existing classes (a bottom-up model) or to generate a class skeleton given existing WSDL (a top-down model).\n</p>\n<ul><li>A developer using a bottom-up model writes implementing classes first (in some programming language) and then uses a WSDL generating tool to expose methods from these classes as a Web service. This is simpler to develop but may be harder to maintain if the original classes are subject to frequent change.<sup id="cite_ref-6" class="reference"><a href="#cite_note-6">&#91;5&#93;</a></sup></li>\n<li>A developer using a top-down model writes the WSDL document first and then uses a code generating tool to produce the class skeleton, to be completed as necessary. This model is generally considered more difficult but can produce cleaner designs and is generally more resistant to change. As long as the message formats between the sender and receiver do not change, changes in the sender and receiver themselves do not affect the Web service. The technique is also referred to as <i>contract first</i> since the WSDL (or contract between sender and receiver) is the starting point.<sup id="cite_ref-7" class="reference"><a href="#cite_note-7">&#91;6&#93;</a></sup></li>\n<li>A developer using a Subset WSDL (SWSDL)<sup id="cite_ref-SWSDL_8-0" class="reference"><a href="#cite_note-SWSDL-8">&#91;7&#93;</a></sup> (i.e. a WSDL with the subset operation in the original WSDL) can perform Web service testing and top-down development.</li></ul>\n<h3><span class="mw-headline" id="Criticism">Criticism</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Web_service&amp;action=edit&amp;section=9" title="Edit section: Criticism">edit</a><span class="mw-editsection-bracket">]</span></span></h3>\n<p>Critics of non-RESTful Web services often complain that they are too complex<sup id="cite_ref-9" class="reference"><a href="#cite_note-9">&#91;8&#93;</a></sup> and based upon large software vendors or integrators, rather than typical <a href="/wiki/Open-source_model" title="Open-source model">open source</a> implementations.\n</p><p>There are also concerns about performance due to Web services\' use of XML as a message format and SOAP/HTTP in enveloping and transporting.<sup id="cite_ref-10" class="reference"><a href="#cite_note-10">&#91;9&#93;</a></sup>\n</p>\n<h3><span class="mw-headline" id="Regression_testing_of_Web_services">Regression testing of Web services</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Web_service&amp;action=edit&amp;section=10" title="Edit section: Regression testing of Web services">edit</a><span class="mw-editsection-bracket">]</span></span></h3>\n<p>Functional and non-functional testing of Web services is done with the help of WSDL parsing. <a href="/wiki/Regression_testing" title="Regression testing">Regression testing</a> is performed by identifying the changes made to upgrade software. Web service regression testing needs can be categorized in three different ways, namely, changes in WSDL, changes in the code, and selective re-testing of operations. We can capture the above three needs in three intermediate forms of Subset WSDL,<sup id="cite_ref-SWSDL_8-1" class="reference"><a href="#cite_note-SWSDL-8">&#91;7&#93;</a></sup> namely, Difference WSDL (DWSDL), Unit WSDL (UWSDL), and Reduced WSDL (RWSDL), respectively. These three Subset WSDLs are then combined to form Combined WSDL (CWSDL) that is further used for regression testing of the Web service. This will help in Automated Web Service Change Management (AWSCM),<sup id="cite_ref-AWSCM_11-0" class="reference"><a href="#cite_note-AWSCM-11">&#91;10&#93;</a></sup> by performing the selection of the relevant test cases to construct a reduced test suite from the old test suite.\n<sup id="cite_ref-12" class="reference"><a href="#cite_note-12">&#91;11&#93;</a></sup>\n</p><p>Web services testing can also be automated using several test automation tools like SOAP UI, <a href="/wiki/Oracle_Application_Testing_Suite" title="Oracle Application Testing Suite">Oracle Application Testing Suite</a> (OATS),<sup id="cite_ref-13" class="reference"><a href="#cite_note-13">&#91;12&#93;</a></sup><sup id="cite_ref-14" class="reference"><a href="#cite_note-14">&#91;13&#93;</a></sup> Unified Functional Testing, Selenium, etc.\n</p>\n<h3><span class="mw-headline" id="Web_service_change_management">Web service change management</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Web_service&amp;action=edit&amp;section=11" title="Edit section: Web service change management">edit</a><span class="mw-editsection-bracket">]</span></span></h3>\n<p>Work-related to the capture and visualization of changes made to a Web service. Visualization and computation of changes can be done in the form of intermediate artifacts (Subset WSDL).<sup id="cite_ref-SWSDL_8-2" class="reference"><a href="#cite_note-SWSDL-8">&#91;7&#93;</a></sup> The insight on the computation of change impact is helpful in testing, top-down development and reduce regression testing. AWSCM<sup id="cite_ref-AWSCM_11-1" class="reference"><a href="#cite_note-AWSCM-11">&#91;10&#93;</a></sup> is a tool that can identify subset operations in a WSDL file to construct a subset WSDL.\n</p>\n<h2><span class="mw-headline" id="See_also">See also</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a
