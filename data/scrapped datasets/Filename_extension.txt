The extension indicates a characteristic of the file contents or its intended use. A filename extension is typically delimited from the filename with a <a href="/wiki/Full_stop" title="Full stop">full stop</a> (period), but in some systems<sup id="cite_ref-1" class="reference"><a href="#cite_note-1">&#91;1&#93;</a></sup> it is separated with spaces.\n</p><p>Some <a href="/wiki/File_system" title="File system">file systems</a> implement filename extensions as a feature of the file system itself and may limit the length and format of the extension, while others treat filename extensions as part of the filename without special distinction.\n</p>\n<div id="toc" class="toc" links</span></a></li>\n</ul>\n</div>\n\n<h2><span class="mw-headline" id="Usage">Usage</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Filename_extension&amp;action=edit&amp;section=1" title="Edit section: Usage">edit</a><span class="mw-editsection-bracket">]</span></span></h2>\n<p>Filename extensions may be considered a type of <a href="/wiki/Metadata" title="Metadata">metadata</a>.<sup id="cite_ref-Stauffer_et_al_2017_2-0" class="reference"><a href="#cite_note-Stauffer_et_al_2017-2">&#91;2&#93;</a></sup> They are commonly used to imply information about the way data might be stored in the file. The exact definition, giving the criteria for deciding what part of the file name is its extension, belongs to the rules of the specific <a href="/wiki/Filesystem" class="mw-redirect" title="Filesystem">filesystem</a> used; usually the extension is the substring which follows the last occurrence, if any, of the <a href="/wiki/Full_stop" title="Full stop">dot character</a> (<i>example:</i> <code>txt</code> is the extension of the filename <code>readme.txt</code>, and <code>html</code> the extension of <code>mysite.index.html</code>).\nOn file systems of some mainframe systems such as <a href="/wiki/Conversational_Monitor_System" title="Conversational Monitor System">CMS</a> in <a href="/wiki/VM_(operating_system)" title="VM (operating system)">VM</a>, <a href="/wiki/OpenVMS" title="OpenVMS">VMS</a>, and of PC systems such as <a href="/wiki/CP/M" title="CP/M">CP/M</a> and derivative systems such as <a href="/wiki/MS-DOS" title="MS-DOS">MS-DOS</a>, the extension is a separate <a href="/wiki/Namespace" title="Namespace">namespace</a> from the filename. Under Microsoft\'s <a href="/wiki/DOS" title="DOS">DOS</a> and <a href="/wiki/Microsoft_Windows" title="Microsoft Windows">Windows</a>, extensions such as <code><a href="/wiki/EXE" class="mw-redirect" title="EXE">EXE</a></code>, <code>COM</code> or <code>BAT</code> indicate that a file is a program <a href="/wiki/Executable" title="Executable">executable</a>. In <a href="/wiki/OS/360_and_successors" title="OS/360 and successors">OS/360 and successors</a>, the part of the dataset name following the last period is treated as an extension by some software, e.g., <a href="/wiki/Time_Sharing_Option" title="Time Sharing Option">TSO</a> EDIT, but it has no special significance to the operating system itself; the same applies to Unix files in MVS.\n</p><p>Filesystems for <a href="/wiki/UNIX-like" class="mw-redirect" title="UNIX-like">UNIX-like</a> operating systems do not separate the extension metadata from the rest of the file name. The <a href="/wiki/Full_stop" title="Full stop">dot character</a> is just another character in the main filename. A file name can have no extensions, a single extension, or more than one extension. More than one extension usually represents nested transformations, such as <code>files.tar.gz</code> (the <code>.tar</code> indicates that the file is a <a href="/wiki/Tar_(computing)" title="Tar (computing)">tar archive</a> of one or more files, and the <code>.gz</code> indicates that the tar archive file is compressed with <a href="/wiki/Gzip" title="Gzip">gzip</a>). Programs transforming or creating files may add the appropriate extension to names inferred from input file names (unless explicitly given an output file name), but programs reading files usually ignore the information; it is mostly intended for the human user. \nIt is more common, especially in binary files, for the file itself to contain <a href="/wiki/File_format#Internal_Metadata" title="File format">internal metadata</a> describing its contents.\nThis model generally requires the full filename to be provided in commands, whereas the metadata approach often allows the extension to be omitted.\n</p><p>The <a href="/wiki/VFAT" class="mw-redirect" title="VFAT">VFAT</a>, <a href="/wiki/NTFS" title="NTFS">NTFS</a>, and <a href="/wiki/ReFS" title="ReFS">ReFS</a> file systems for <a href="/wiki/Windows" class="mw-redirect" title="Windows">Windows</a> also do not separate the extension metadata from the rest of the file name, and allow multiple extensions.\n</p><p>With the advent of <a href="/wiki/Graphical_user_interface" title="Graphical user interface">graphical user interfaces</a>, the issue of file management and interface behavior arose. Microsoft Windows allowed multiple applications to be associated with a given extension, and different actions were available for selecting the required application, such as a <a href="/wiki/Context_menu" title="Context menu">context menu</a> offering a choice between viewing, editing or printing the file. The assumption was still that any extension represented a single file type; there was an unambiguous mapping between extension and icon.\n</p><p>The <a href="/wiki/Classic_Mac_OS" title="Classic Mac OS">classic Mac OS</a> disposed of filename-based extension metadata entirely; it used, instead, a distinct file <a href="/wiki/Type_code" class="mw-redirect" title="Type code">type code</a> to identify the file format. Additionally, a <a href="/wiki/Creator_code" title="Creator code">creator code</a> was specified to determine which application would be launched when the file\'s <a href="/wiki/Icon_(computing)" title="Icon (computing)">icon</a> was <a href="/wiki/Double-click" title="Double-click">double-clicked</a>. <a href="/wiki/MacOS" title="MacOS">macOS</a>, however, uses filename suffixes, as well as type and creator codes, as a consequence of being derived from the UNIX-like <a href="/wiki/NeXTSTEP" title="NeXTSTEP">NeXTSTEP</a> operating system.\n</p>\n<h2><span class="mw-headline" id="Improvements">Improvements</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Filename_extension&amp;action=edit&amp;section=2" title="Edit section: Improvements">edit</a><span class="mw-editsection-bracket">]</span></span></h2>\n<p>The filename extension was originally used to determine the file\'s generic type.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">&#91;<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (November 2015)">citation needed</span></a></i>&#93;</sup> The need to condense a file\'s type into <a href="/wiki/8.3_filename" title="8.3 filename">three characters</a> frequently led to abbreviated extensions. Examples include using <code>.GFX</code> for graphics files, <code>.TXT</code> for <a href="/wiki/Plain_text" title="Plain text">plain text</a>, and <code>.MUS</code> for music. However, because many different software programs have been made that all handle these data types (and others) in a variety of ways, filename extensions started to become closely associated with certain products\xe2\x80\x94even specific product versions. For example, early <a href="/wiki/WordStar" title="WordStar">WordStar</a> files used <code>.WS</code> or <code>.WS<i>n</i></code>, where <i>n</i> was the program\'s version number. Also, conflicting uses of some filename extensions developed. One example is <code>.rpm</code>, used for both <a href="/wiki/RPM_Package_Manager" title="RPM Package Manager">RPM Package Manager</a> packages and <a href="/wiki/RealPlayer" title="RealPlayer">RealPlayer</a> Media files;.<sup id="cite_ref-3" class="reference"><a href="#cite_note-3">&#91;3&#93;</a></sup> Others are <code>.qif</code>, shared by <a href="/wiki/DESQview" title="DESQview">DESQview</a> fonts, <a href="/wiki/Quicken" title="Quicken">Quicken</a> financial ledgers, and <a href="/wiki/QuickTime" title="QuickTime">QuickTime</a> pictures;<sup id="cite_ref-4" class="reference"><a href="#cite_note-4">&#91;4&#93;</a></sup> <code>.gba</code>, shared by <a href="/wiki/GrabIt" title="GrabIt">GrabIt</a> scripts and <a href="/wiki/Game_Boy_Advance" title="Game Boy Advance">Game Boy Advance</a> ROM images;<sup id="cite_ref-5" class="reference"><a href="#cite_note-5">&#91;5&#93;</a></sup> <code>.sb</code>, used for <a href="/wiki/SmallBasic" class="mw-redirect" title="SmallBasic">SmallBasic</a> and <a href="/wiki/Scratch_(programming_language)" title="Scratch (programming language)">Scratch</a>; and <code>.dts</code>, being used for <a href="/wiki/Torque_Game_Engine" class="mw-redirect" title="Torque Game Engine">Dynamix Three Space</a> and <a href="/wiki/DTS_(sound_system)" title="DTS (sound system)">DTS</a>.\n</p><p>Some other operating systems that used filename extensions generally had fewer restrictions on filenames. Many allowed full filename lengths of 14 or more characters, and maximum name lengths up to 255 were not uncommon. The file systems in operating systems such as <a href="/wiki/Multics" title="Multics">Multics</a> and <a href="/wiki/UNIX" class="mw-redirect" title="UNIX">UNIX</a> stored the file name as a single string, not split into base name and extension components, allowing the "." to be just another character allowed in file names. Such systems generally allow for variable-length filenames, permitting more than one dot, and hence multiple suffixes. Some components of Multics and UNIX, and applications running on them, used suffixes, in some cases, to indicate file types, but they did not use them as much\xe2\x80\x94for example, executables and ordinary text files had no suffixes in their names.\n</p><p>The <a href="/wiki/High_Performance_File_System" title="High Performance File System">High Performance File System</a> (HPFS), used in Microsoft and <a href="/wiki/IBM" title="IBM">IBM</a>\'s <a href="/wiki/OS/2" title="OS/2">OS/2</a> also supported long file names and did not divide the file name into a name and an extension. The convention of using suffixes continued, even though HPFS supported extended attributes for files, allowing a file\'s type to be stored in the file as an extended attribute.\n</p><p>Microsoft\'s <a href="/wiki/Windows_NT" title="Windows NT">Windows NT</a>\'s native file system, <a href="/wiki/NTFS" title="NTFS">NTFS</a>, supported long file names and did not divide the file name into a name and an extension, but again, the convention of using suffixes to simulate extensions continued, for compatibility with existing versions of Windows.\n</p><p>When the <a href="/wiki/Internet" title="Internet">Internet</a> age first arrived, those using Windows systems that were still restricted to 8.3 filename formats had to create web pages with names ending in <code>.HTM</code>, while those using <a href="/wiki/Macintosh" title="Macintosh">Macintosh</a> or UNIX computers could use the recommended <code>.html</code> filename extension. This also became a problem for programmers experimenting with the <a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java programming language</a>, since it <i>requires</i> <a href="/wiki/Source_code" title="Source code">source code</a> files to have the four-letter suffix <code>.java</code> and <a href="/wiki/Compiler" title="Compiler">compiles</a> <a href="/wiki/Object_code" title="Object code">object code</a> output files with the five-letter <code>.class</code> suffix.<sup id="cite_ref-6" class="reference"><a href="#cite_note-6">&#91;6&#93;</a></sup>\n</p><p>Eventually, <a href="/wiki/Windows_95" title="Windows 95">Windows 95</a> introduced support for long file names, and removed the 8.3 name/extension split in file names from non-NT Windows, in an extended version of the commonly used <a href="/wiki/File_Allocation_Table" title="File Allocation Table">FAT</a> <a href="/wiki/File_system" title="File system">file system</a> called <a href="/wiki/VFAT" class="mw-redirect" title="VFAT">VFAT</a>. VFAT first appeared in <a href="/wiki/Windows_NT_3.5" title="Windows NT 3.5">Windows NT 3.5</a> and <a href="/wiki/Windows_95" title="Windows 95">Windows 95</a>. The internal implementation of long file names in VFAT is largely considered to be a <a href="/wiki/Kludge" title="Kludge">kludge</a><sup class="noprint Inline-Template" style="white-space:nowrap;">&#91;<i><a href="/wiki/Wikipedia:Manual_of_Style/Words_to_watch#Unsupported_attributions" title="Wikipedia:Manual of Style/Words to watch"><span title="The material near this tag may use weasel words or too-vague attribution. (October 2018)">by whom?</span></a></i>&#93;</sup>, but it removed the important length restriction and allowed files to have a mix of <a href="/wiki/Upper_case" class="mw-redirect" title="Upper case">upper case</a> and <a href="/wiki/Lower_case" class="mw-redirect" title="Lower case">lower case</a> letters, on machines that would not run <a href="/wiki/Windows_NT" title="Windows NT">Windows NT</a> well.\n</p>\n<h2><span class="mw-headline" id="Command_name_issues">Command name issues</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Filename_extension&amp;action=edit&amp;section=3" title="Edit section: Command name issues">edit</a><span class="mw-editsection-bracket">]</span></span></h2>\n<table class="box-Cleanup plainlinks metadata ambox ambox-style ambox-Cleanup" role="presentation"><tbody><tr><td class="mbox-image"><div style="width:52px"><img alt="" src="//upload.wikimedia.org/wikipedia/en/thumb/f/f2/Edit-clear.svg/40px-Edit-clear.svg.png" decoding="async" width="40" height="40" srcset="//upload.wikimedia.org/wikipedia/en/thumb/f/f2/Edit-clear.svg/60px-Edit-clear.svg.png 1.5x, //upload.wikimedia.org/wikipedia/en/thumb/f/f2/Edit-clear.svg/80px-Edit-clear.svg.png 2x" data-file-width="48" data-file-height="48" /></div></td><td class="mbox-text"><div class="mbox-text-span">This section may <b>require <a href="/wiki/Wikipedia:Cleanup" title="Wikipedia:Cleanup">cleanup</a></b> to meet Wikipedia\'s <a href="/wiki/Wikipedia:Manual_of_Style" title="Wikipedia:Manual of Style">quality standards</a>. The specific problem is: <b>intractable construction</b><span class="hide-when-compact"> Please help <a class="external text" href="https://en.wikipedia.org/w/index.php?title=Filename_extension&amp;action=edit">improve this section</a> if you can.</span>  <small class="date-container"><i>(<span class="date">November 2015</span>)</i></small><small class="hide-when-compact"><i> (<a href="/wiki/Help:Maintenance_template_removal" title="Help:Maintenance template removal">Learn how and when to remove this template message</a>)</i></small></div></td></tr></tbody></table>\n<p>The use of a filename extension in a command name appears occasionally, usually as a side effect of the command having been implemented as a script, e.g., for the <a href="/wiki/Bourne_shell" title="Bourne shell">Bourne shell</a> or for <a href="/wiki/Python_(programming_language)" title="Python (programming language)">Python</a>, and the interpreter name being suffixed to the command name, a practice common on systems that rely on associations between filename extension and interpreter, but sharply deprecated<sup id="cite_ref-7" class="reference"><a href="#cite_note-7">&#91;7&#93;</a></sup> in <a href="/wiki/UNIX" class="mw-redirect" title="UNIX">UNIX</a>-derived systems like <a href="/wiki/Linux" title="Linux">Linux</a> and Apple\'s <a href="/wiki/MacOS" title="MacOS">macOS</a>, where the interpreter is normally specified as a header in the script ("<a href="/wiki/Shebang_(Unix)" title="Shebang (Unix)">shebang</a>").\n</p><p>On association-based systems, the filename extension is generally mapped to a single, system-wide selection of interpreter for that extension (such as ".py" meaning to use Python), and the command itself is runnable from the command line even if the extension is omitted (assuming appropriate setup is done). If the implementation language is changed, the command name extension is changed as well, and the OS provides a consistent <a href="/wiki/API" title="API">API</a> by allowing the same extension-less version of the command to be used in both cases. This method suffers somewhat from the essentially global nature of the association mapping, as well as from developers\' incomplete avoidance of extensions when calling programs, and that developers can\'t force that avoidance. Windows is the only remaining widespread employer of this mechanism.\n</p><p>On systems with <a href="/wiki/Interpreter_directive" title="Interpreter directive">interpreter directives</a>, including virtually all versions of Unix, command name extensions have no special significance, and are by standard practice not used, since the primary method to set interpreters for scripts is to start them with a single line specifying the interpreter to use (which could be viewed as a degenerate <a href="/wiki/Resource_fork" title="Resource fork">resource fork</a>). In these environments, including the extension in a command name unnecessarily exposes an implementation detail which puts all references to the commands from other programs at future risk if the implementation changes. For example, it would be perfectly normal for a shell script to be reimplemented in Python or Ruby, and later in C or C++, all of which would change the name of the command were extensions used. Without extensions, a program always has the same extension-less name, with only the <a href="/wiki/Interpreter_directive" title="Interpreter directive">interpreter directive</a> and/or <a href="/wiki/Magic_number_(programming)" title="Magic number (programming)">magic number</a> changing, and references to the program from other programs remain valid.\n</p>\n<h2><span class="mw-headline" id="Security_issues">Security issues</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Filename_extension&amp;action=edit&amp;section=4" title="Edit section: Security issues">edit</a><span class="mw-editsection-bracket">]</span></span></h2>\n<p>The default behavior of <a href="/wiki/File_Explorer" title="File Explorer">File Explorer</a>, the file browser provided with <a href="/wiki/Microsoft_Windows" title="Microsoft Windows">Microsoft Windows</a>, is for filename extensions to not be displayed. Malicious users have tried to spread <a href="/wiki/Computer_virus" title="Computer virus"> computer viruses</a> and <a href="/wiki/Computer_worm" title="Computer worm"> computer worms</a> by using file names formed like <a href="/wiki/ILOVEYOU" title="ILOVEYOU"><code>LOVE-LETTER-FOR-YOU.TXT.vbs</code></a>. The hope is that this will appear as <code>LOVE-LETTER-FOR-YOU.TXT</code>, a harmless text file, without alerting the user to the fact that it is a harmful computer program, in this case, written in <a href="/wiki/VBScript" title="VBScript">VBScript</a>. Default behavior for <a href="/wiki/ReactOS" title="ReactOS">ReactOS</a> is to display filename extensions in <a href="/wiki/Windows_Explorer" class="mw-redirect" title="Windows Explorer">ReactOS Explorer</a>.\n</p><p>Later Windows versions (starting with <a href="/wiki/Windows_XP" title="Windows XP">Windows XP</a> <a href="/wiki/XP_Service_Pack_2" class="mw-redirect" title="XP Service Pack 2">Service Pack 2</a> and <a href="/wiki/Windows_Server_2003" title="Windows Server 2003">Windows Server 2003</a>) included customizable lists of filename extensions that should be considered "dangerous" in certain "zones" of operation, such as when <a href="/wiki/Download" title="Download">downloaded</a> from the <a href="/wiki/World_Wide_Web" title="World Wide Web">web</a> or received as an e-mail attachment. Modern <a href="/wiki/Antivirus_software" title="Antivirus software">antivirus software</a> systems also help to defend users against such attempted attacks where possible.\n</p><p>Some viruses take advantage of the similarity between the "<a href="/wiki/.com" title=".com">.com</a>" <a href="/wiki/Top-level_domain" title="Top-level domain">top-level domain</a> and the <a href="/wiki/COM_file" title="COM file">".COM" filename extension</a> by emailing malicious, executable command-file attachments under names superficially similar to URLs (<i>e.g.</i>, "myparty.yahoo.com"), with the effect that some naive users click on email-embedded links that they think lead to websites but actually download and execute the malicious attachments.\n</p><p>There have been instances of <a href="/wiki/Malware" title="Malware">malware</a> crafted to exploit vulnerabilities in some Windows applications which could cause a stack-based <a href="/wiki/Buffer_overflow" title="Buffer overflow">buffer overflow</a> when opening a file with an overly long, unhandled filename extension.\n</p><p>The filename extension is just a marker and the content of the file does not have to match it.<sup id="cite_ref-8" class="reference"><a href="#cite_note-8">&#91;8&#93;</a></sup> This can be used to disguise malicious content. When trying to identify a file for security reasons, it is therefore considered dangerous to rely on the extension alone and a proper analysis of the content of the file is preferred. For example, on <a href="/wiki/UNIX" class="mw-redirect" title="UNIX">UNIX</a> derived systems, it is not uncommon to find files with no extensions at all, as commands such as <a href="/wiki/File_(command)" title="File (command)">file (command)</a> are meant to be used instead, and will read the file\'s header to determine its content.\n</p>\n<h2><span class="mw-headline" id="Alternatives">Alternatives</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Filename_extension&amp;action=edit&amp;section=5" title="Edit section: Alternatives">edit</a><span class="mw-editsection-bracket">]</span></span></h2>\n<p>In many <a href="/wiki/Internet_protocol_suite" title="Internet protocol suite">Internet</a> protocols, such as <a href="/wiki/HTTP_protocol" class="mw-redirect" title="HTTP protocol">HTTP</a> and <a href="/wiki/MIME" title="MIME">MIME email</a>, the type of a bitstream is stated as the <a href="/wiki/Media_type" title="Media type">media type</a>, or MIME type, of the stream, rather than a filename extension. This is given in a line of text preceding the stream, such as <i>Content-type: text/plain</i>.\n</p><p>There is no standard mapping between filename extensions and media types, resulting in possible mismatches in interpretation between authors, web servers, and client software when transferring files over the Internet. For instance, a content author may specify the extension <i>svgz</i> for a compressed <a href="/wiki/Scalable_Vector_Graphics" title="Scalable Vector Graphics">Scalable Vector Graphics</a> file, but a web server that does not recognize this extension may not send the proper content type <i>application/svg+xml</i> and its required compression header, leaving web browsers unable to correctly interpret and display the image.\n</p><p><a href="/wiki/BeOS" title="BeOS">BeOS</a>, whose <a href="/wiki/Be_File_System" title="Be File System">BFS</a> file system supports extended attributes, would tag a file with its media type as an extended attribute. The <a href="/wiki/KDE" title="KDE">KDE</a> and <a href="/wiki/GNOME" title="GNOME">GNOME</a> <a href="/wiki/Desktop_environment" title="Desktop environment">desktop environments</a> associate a media type with a file by examining both the filename suffix and the contents of the file, in the fashion of the <a href="/wiki/File_(command)" title="File (command)">file</a> command, as a <a href="/wiki/Heuristic" title="Heuristic">heuristic</a>. They choose the application to launch when a file is opened based on that media type, reducing the dependency on filename extensions. <a href="/wiki/MacOS" title="MacOS">macOS</a> uses both filename extensions and media types, as well as <a href="/wiki/OSType" class="mw-redirect" title="OSType">file type codes</a>, to select a <a href="/wiki/Uniform_Type_Identifier" title="Uniform Type Identifier">Uniform Type Identifier</a> by which to identify the file type internally.\n</p>\n<h2><span class="mw-headline" id="See_also">See also</span><span class="mw-editsection"><span
