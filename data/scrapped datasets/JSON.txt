class="IPA nopopups noexcerpt"><a href="/wiki/Help:IPA/English" title="Help:IPA/English">/<span style="border-bottom:1px dotted"><span title="/\xcb\x88/: primary stress follows">\xcb\x88</span><span title="/\xc9\x99n/: &#39;on&#39; in &#39;button&#39;">\xc9\x99n</span></span>/</a></span></span>; also <span class="rt-commentedText nowrap"><span class="IPA nopopups noexcerpt"><a href="/wiki/Help:IPA/English" title="Help:IPA/English">/<span style="border-bottom:1px dotted"><span title="/\xcb\x88/: primary stress follows">\xcb\x88</span><span title="/d\xca\x92/: &#39;j&#39; in &#39;jam&#39;">d\xca\x92</span><span title="/e\xc9\xaa/: &#39;a&#39; in &#39;face&#39;">e\xc9\xaa</span><span title="/\xcb\x8c/: secondary stress follows">\xcb\x8c</span><span title="&#39;s&#39; in &#39;sigh&#39;">s</span><span title="/\xc9\x92/: &#39;o&#39; in &#39;body&#39;">\xc9\x92</span><span title="&#39;n&#39; in &#39;nigh&#39;">n</span></span>/</a></span></span>) is an <a data interchange">data interchange</a> format that uses <a href="/wiki/Human-readable_medium" title="Human-readable medium">human-readable</a> text to store and transmit data objects consisting of <a href="/wiki/Attribute%E2%80%93value_pair" title="Attribute\xe2\x80\x93value pair">attribute\xe2\x80\x93value pairs</a> and <a href="/wiki/Array_data_type" title="Array data type">arrays</a> (or other <a href="/wiki/Serialization" title="Serialization">serializable</a> values). It is a very common <a href="/wiki/Data" title="Data">data</a> format, with a diverse range of applications, one example being <a href="/wiki/Web_applications" class="mw-redirect" title="Web applications">web applications</a> that communicate with a <a href="/wiki/Server_(computing)" title="Server (computing)">server</a>.\n</p><p>JSON is a <a href="/wiki/Language-independent_specification" title="Language-independent specification">language-independent</a> data format. It was derived from <a href="/wiki/JavaScript" title="JavaScript">JavaScript</a>, but many modern <a href="/wiki/Programming_language" title="Programming language">programming languages</a> include code to generate and <a href="/wiki/Parsing" title="Parsing">parse</a> JSON-format data. JSON filenames use the extension <code>.json</code>.\n</p><p><a href="/wiki/Douglas_Crockford" title="Douglas Crockford">Douglas Crockford</a> originally specified the JSON format in the early (2007)</div></div></div>\n<p>JSON grew out of a need for <a href="/wiki/State_(computer_science)" title="State (computer science)">stateless</a>, real-time server-to-browser communication protocol without using browser plugins such as <a href="/wiki/Adobe_Flash" title="Adobe Flash">Flash</a> or <a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a> applets, the dominant methods used in the early 2000s.<sup id="cite_ref-Edu4java,_2014_9-0" class="reference"><a href="#cite_note-Edu4java,_2014-9">&#91;9&#93;</a></sup>\n</p><p>A precursor to the JSON libraries was used in a children\'s digital asset trading game project named <a href="/wiki/Cartoon_Orbit" title="Cartoon Orbit">Cartoon Orbit</a> at Communities.com (at which State Software\'s co-founders had all worked previously) for Cartoon Network, which used a browser side plug-in with a proprietary messaging format to manipulate <a href="/wiki/Dynamic_HTML" title="Dynamic HTML">Dynamic HTML</a> elements (this system is also owned by 3DO). Upon discovery of early <a href="/wiki/Ajax_(programming)" title="Ajax (programming)">Ajax</a> capabilities, digiGroups, Noosh, and others used frames to pass information into the user browsers\' visual field without refreshing a Web application\'s visual context, realizing real-time rich Web applications using only the standard HTTP, HTML and JavaScript capabilities of Netscape 4.0.5+ and IE 5+.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">&#91;<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (December 2020)">citation needed</span></a></i>&#93;</sup>\n</p><p>Crockford first specified and popularized the JSON format.<sup id="cite_ref-10" class="reference"><a href="#cite_note-10">&#91;10&#93;</a></sup> The State Software co-founders agreed to build a system that used standard browser capabilities and provided an <a href="/wiki/Abstraction_layer" title="Abstraction layer">abstraction layer</a> for Web developers to create stateful Web applications that had a persistent duplex connection to a Web server by holding two <a href="/wiki/Hypertext_Transfer_Protocol" title="Hypertext Transfer Protocol">Hypertext Transfer Protocol</a> (HTTP) connections open and recycling them before standard browser time-outs if no further data were exchanged. The co-founders had a round-table discussion and voted whether to call the data format JSML or JSON, as well as under what <a href="/wiki/Software_license" title="Software license">license</a> type to make it available. <a href="/wiki/Chip_Morningstar" title="Chip Morningstar">Chip Morningstar</a> developed the idea for the State Application Framework at State Software.<sup id="cite_ref-Morningstar_11-0" class="reference"><a href="#cite_note-Morningstar-11">&#91;11&#93;</a></sup><sup id="cite_ref-State_Software_12-0" class="reference"><a href="#cite_note-State_Software-12">&#91;12&#93;</a></sup>\n</p><p>The system was sold to <a href="/wiki/Sun_Microsystems" title="Sun Microsystems">Sun Microsystems</a>, <a href="/wiki/Amazon.com" class="mw-redirect" title="Amazon.com">Amazon.com</a> and <a href="/wiki/Electronic_Data_Systems" title="Electronic Data Systems">EDS</a>. The JSON.org<sup id="cite_ref-13" class="reference"><a href="#cite_note-13">&#91;13&#93;</a></sup> website was launched in 2002. In December 2005, <a href="/wiki/Yahoo!" title="Yahoo!">Yahoo!</a> began offering some of its <a href="/wiki/Web_service" title="Web service">Web services</a> in JSON.<sup id="cite_ref-yahoo_14-0" class="reference"><a href="#cite_note-yahoo-14">&#91;14&#93;</a></sup>\n</p><p>JSON was based on a <a href="/wiki/Subset" title="Subset">subset</a> of the <a href="/wiki/JavaScript" title="JavaScript">JavaScript</a> scripting language (specifically, Standard <a href="/wiki/Ecma_International" title="Ecma International">ECMA</a>-262 3rd Edition\xe2\x80\x94December 1999<sup id="cite_ref-15" class="reference"><a href="#cite_note-15">&#91;15&#93;</a></sup>) and is commonly used with JavaScript, but it is a <a href="/wiki/Language-independent_specification" title="Language-independent specification">language-independent</a> data format. Code for <a href="/wiki/Parsing" title="Parsing">parsing</a> and generating JSON data is readily available in many <a href="/wiki/Programming_languages" class="mw-redirect" title="Programming languages">programming languages</a>. JSON\'s website lists JSON <a href="/wiki/Language_binding" title="Language binding">libraries</a> by language.\n</p><p>In October 2013, <a href="/wiki/Ecma_International" title="Ecma International">Ecma International</a> published the first edition of its JSON standard ECMA-404.<sup id="cite_ref-ecma2013_6-1" class="reference"><a href="#cite_note-ecma2013-6">&#91;6&#93;</a></sup> That same year, <link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r999302996"/><a href="/wiki/RFC_(identifier)" class="mw-redirect" title="RFC (identifier)">RFC</a>&#160;<a rel="nofollow" class="external text" href="https://tools.ietf.org/html/rfc7158">7158</a> used ECMA-404 as a reference. In 2014, <link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r999302996"/><a href="/wiki/RFC_(identifier)" class="mw-redirect" title="RFC (identifier)">RFC</a>&#160;<a rel="nofollow" class="external text" href="https://tools.ietf.org/html/rfc7159">7159</a> became the main reference for JSON\'s Internet uses, superseding <link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r999302996"/><a href="/wiki/RFC_(identifier)" class="mw-redirect" title="RFC (identifier)">RFC</a>&#160;<a rel="nofollow" class="external text" href="https://tools.ietf.org/html/rfc4627">4627</a> and <link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r999302996"/><a href="/wiki/RFC_(identifier)" class="mw-redirect" title="RFC (identifier)">RFC</a>&#160;<a rel="nofollow" class="external text" href="https://tools.ietf.org/html/rfc7158">7158</a> (but preserving ECMA-262 and ECMA-404 as main references). In November 2017, <a href="/wiki/ISO/IEC_JTC_1/SC_22" title="ISO/IEC JTC 1/SC 22">ISO/IEC JTC 1/SC 22</a> published ISO/IEC 21778:2017<sup id="cite_ref-:0_1-2" class="reference"><a href="#cite_note-:0-1">&#91;1&#93;</a></sup> as an international standard. On 13 December 2017, the <a href="/wiki/Internet_Engineering_Task_Force" title="Internet Engineering Task Force">Internet Engineering Task Force</a> obsoleted <link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r999302996"/><a href="/wiki/RFC_(identifier)" class="mw-redirect" title="RFC (identifier)">RFC</a>&#160;<a rel="nofollow" class="external text" href="https://tools.ietf.org/html/rfc7159">7159</a> when it published <link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r999302996"/><a href="/wiki/RFC_(identifier)" class="mw-redirect" title="RFC (identifier)">RFC</a>&#160;<a rel="nofollow" class="external text" href="https://tools.ietf.org/html/rfc8259">8259</a>, which is the current version of the <a href="/wiki/Internet_Standard" title="Internet Standard">Internet Standard</a> STD 90.<sup id="cite_ref-16" class="reference"><a href="#cite_note-16">&#91;16&#93;</a></sup><sup id="cite_ref-17" class="reference"><a href="#cite_note-17">&#91;17&#93;</a></sup>\n</p><p>Crockford added a clause to the JSON license stating that "The Software shall be used for Good, not Evil," in order to <a href="/wiki/Open-source_model" title="Open-source model">open-source</a> the JSON libraries while mocking corporate lawyers and those who are overly pedantic. On the other hand, this clause led to <a href="/wiki/License_compatibility" title="License compatibility">license compatibility</a> problems of the JSON license with other <a href="/wiki/Open-source_license" title="Open-source license">open-source licenses</a>, as <a href="/wiki/Open-source_software" title="Open-source software">open-source software</a> and <a href="/wiki/Free_software" title="Free software">free software</a> usually imply no restrictions on the purpose of use.<sup id="cite_ref-18" class="reference"><a href="#cite_note-18">&#91;18&#93;</a></sup>\n</p>\n<h2><span class="mw-headline" id="Syntax">Syntax</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JSON&amp;action=edit&amp;section=5" title="Edit section: Character encoding">edit</a><span class="mw-editsection-bracket">]</span></span></h3>\n<p>Although Crockford originally asserted and believed that JSON is a strict subset of JavaScript and ECMAScript,<sup id="cite_ref-19" class="reference"><a href="#cite_note-19">&#91;19&#93;</a></sup> his specification actually allows valid JSON documents that are not valid JavaScript; JSON allows the <a href="/wiki/Unicode#Newlines" title="Unicode">Unicode line terminators</a> <span class="nowrap"><link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r886049734"/><span class="monospaced">U+2028</span>&#x20;</span> <span style="font-size:85%;">LINE SEPARATOR</span> and <span class="nowrap"><link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r886049734"/><span class="monospaced">U+2029</span>&#x20;</span> <span style="font-size:85%;">PARAGRAPH SEPARATOR</span> to appear unescaped in quoted strings, while ECMAScript 2018 and older does not.<sup id="cite_ref-json-2028_20-0" class="reference"><a href="#cite_note-json-2028-20">&#91;20&#93;</a></sup><sup id="cite_ref-21" class="reference"><a href="#cite_note-21">&#91;21&#93;</a></sup> This is a consequence of JSON disallowing only "control characters". For maximum portability, these characters should be backslash-escaped. This subtlety is important when generating <a href="/wiki/JSONP" title="JSONP">JSONP</a>.<sup class="noprint Inline-Template" style="white-space:nowrap;">&#91;<i><a href="/wiki/Wikipedia:No_original_research" title="Wikipedia:No original research"><span title="The material near this tag possibly contains original research. (December 2020)">original research?</span></a></i>&#93;</sup>\n</p><p>JSON exchange in an open ecosystem must be encoded in <a href="/wiki/UTF-8" title="UTF-8">UTF-8</a>.<sup id="cite_ref-rfc8259_7-1" class="reference"><a href="#cite_note-rfc8259-7">&#91;7&#93;</a></sup> The encoding supports the full Unicode character set, including those characters outside the <a href="/wiki/Plane_(Unicode)#Basic_Multilingual_Plane" title="Plane (Unicode)">Basic Multilingual Plane</a> (U+10000 to U+10FFFF). However, if escaped, those characters must be written using <a href="/wiki/UTF-16" title="UTF-16">UTF-16</a> surrogate pairs, a detail missed by some JSON parsers.<sup class="noprint Inline-Template" style="white-space:nowrap;">&#91;<i><a href="/wiki/Wikipedia:No_original_research" title="Wikipedia:No original research"><span title="The material near this tag possibly contains original research. (December 2020)">original research?</span></a></i>&#93;</sup> For example, to include the <a href="/wiki/Emoji" title="Emoji">Emoji</a> character <span class="nowrap"><link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r886049734"/><span class="monospaced">U+1F610</span>&#x20;</span><span style="font-size:125%;line-height:1em">&#x1f610;</span> <span style="font-size:85%;">NEUTRAL FACE</span> in JSON:\n</p>\n<div class="mw-highlight mw-highlight-lang-javascript mw-content-ltr" dir="ltr"><pre><span></span><span class="p">{</span> <span class="s2">&quot;face&quot;</span><span class="o">:</span> <span class="s2">&quot;\xf0\x9f\x98\x90&quot;</span> <span class="p">}</span>\n<span class="c1">// or</span>\n<span class="p">{</span> <span class="s2">&quot;face&quot;</span><span class="o">:</span> <span class="s2">&quot;\\uD83D\\uDE10&quot;</span> <span class="p">}</span>\n</pre></div>\n<p>JSON became a strict subset of ECMAScript as of the language\'s 2019 revision.<sup id="cite_ref-ECMATC39_22-0" class="reference"><a href="#cite_note-ECMATC39-22">&#91;22&#93;</a></sup><sup id="cite_ref-ECMATC39Stage4_23-0" class="reference"><a href="#cite_note-ECMATC39Stage4-23">&#91;23&#93;</a></sup>\n</p>\n<h3><span class="mw-headline" id="Data_types">Data types</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JSON&amp;action=edit&amp;section=6" title="Edit section: Data types">edit</a><span class="mw-editsection-bracket">]</span></span></h3>\n<p>JSON\'s basic data types are:\n</p>\n<ul><li>Number: a signed decimal number that may contain a fractional part and may use exponential <a href="/wiki/E_notation" class="mw-redirect" title="E notation">E notation</a>, but cannot include non-numbers such as <a href="/wiki/NaN" title="NaN">NaN</a>. The format makes no distinction between integer and floating-point. JavaScript uses a <a href="/wiki/Double-precision_floating-point_format" title="Double-precision floating-point format">double-precision floating-point format</a> for all its numeric values (until later also supports <a href="/wiki/Arbitrary-precision_arithmetic" title="Arbitrary-precision arithmetic">BigInt</a><sup id="cite_ref-24" class="reference"><a href="#cite_note-24">&#91;24&#93;</a></sup>), but other languages implementing JSON may encode numbers differently.</li>\n<li><a href="/wiki/String_(computer_science)" title="String (computer science)">String</a>: a sequence of zero or more <a href="/wiki/Unicode" title="Unicode">Unicode</a> characters. Strings are delimited with double-quotation marks and support a backslash <a href="/wiki/Escape_character" title="Escape character">escaping</a> syntax.</li>\n<li><a href="/wiki/Boolean_datatype" class="mw-redirect" title="Boolean datatype">Boolean</a>: either of the values <code>true</code> or <code>false</code></li>\n<li><a href="/wiki/Array_data_structure" title="Array data structure">Array</a>: an <a href="/wiki/List_(abstract_data_type)" title="List (abstract data type)">ordered list</a> of zero or more elements, each of which may be of any type. Arrays use <a href="/wiki/Square_bracket" class="mw-redirect" title="Square bracket">square bracket</a> notation with comma-separated elements.</li>\n<li><a href="/wiki/Object_(computer_science)" title="Object (computer science)">Object</a>: a collection of <a href="/wiki/Attribute%E2%80%93value_pair" title="Attribute\xe2\x80\x93value pair">name\xe2\x80\x93value pairs</a> where the names (also called keys) are strings. Objects are intended to represent <a href="/wiki/Associative_array" title="Associative array">associative arrays</a>,<sup id="cite_ref-ecma2013_6-2" class="reference"><a href="#cite_note-ecma2013-6">&#91;6&#93;</a></sup> where each key is unique within an object. Objects are delimited with <a href="/wiki/Braces_(punctuation)#Braces" class="mw-redirect" title="Braces (punctuation)">curly brackets</a> and use commas to separate each pair, while within each pair the colon \':\' character separates the key or name from its value.</li>\n<li><code><a href="/wiki/Nullable_type" title="Nullable type">null</a></code>: an empty value, using the word <code>null</code></li></ul>\n<p><a href="/wiki/Whitespace_character" title="Whitespace character">Whitespace</a> is allowed and ignored around or between syntactic elements (values and punctuation, but not within a string value). Four specific characters are considered whitespace for this purpose: <a href="/wiki/Space_(punctuation)" title="Space (punctuation)">space</a>, <a href="/wiki/Horizontal_tab" class="mw-redirect" title="Horizontal tab">horizontal tab</a>, <a href="/wiki/Line_feed" class="mw-redirect" title="Line feed">line feed</a>, and <a href="/wiki/Carriage_return" title="Carriage return">carriage return</a>. In particular, the <a href="/wiki/Byte_order_mark" title="Byte order mark">byte order mark</a> must not be generated by a conforming implementation (though it may be accepted when parsing JSON). JSON does not provide syntax for <a href="/wiki/Comment_(computer_programming)" title="Comment (computer programming)">comments</a>.<sup id="cite_ref-DouglasCrockfordComments_25-0" class="reference"><a href="#cite_note-DouglasCrockfordComments-25">&#91;25&#93;</a></sup>\n</p><p>Early versions of JSON (such as specified by <link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r999302996"/><a href="/wiki/RFC_(identifier)" class="mw-redirect" title="RFC (identifier)">RFC</a>&#160;<a rel="nofollow" class="external text" href="https://tools.ietf.org/html/rfc4627">4627</a>) required that a valid JSON text must consist of only an object or an array type, which could contain other types within them. This restriction was dropped in <link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r999302996"/><a href="/wiki/RFC_(identifier)" class="mw-redirect" title="RFC (identifier)">RFC</a>&#160;<a rel="nofollow" class="external text" href="https://tools.ietf.org/html/rfc7158">7158</a>, where a JSON text was redefined as any serialized value.\n</p><p>Numbers in JSON are agnostic with regard to their representation within programming languages. While this allows for numbers of <a href="/wiki/Arbitrary-precision_arithmetic" title="Arbitrary-precision arithmetic">arbitrary precision</a> to be serialized, it may lead to portability issues. For example, since no differentiation is made between integer and floating-point values, some implementations may treat <code>42</code>, <code>42.0</code>, and <code>4.2E+1</code> as the same number, while others may not. The JSON standard makes no requirements regarding implementation details such as <a href="/wiki/Arithmetic_overflow" class="mw-redirect" title="Arithmetic overflow">overflow</a>, <a href="/wiki/Arithmetic_underflow" title="Arithmetic underflow">underflow</a>, loss of precision, rounding, or <a href="/wiki/Signed_zeros" class="mw-redirect" title="Signed zeros">signed zeros</a>, but it does recommend to expect no more than <a href="/wiki/IEEE_754" title="IEEE 754">IEEE 754</a> <a href="/wiki/Double-precision_floating-point_format" title="Double-precision floating-point format">binary64</a> precision for "good interoperability". There is no inherent precision loss in serializing a machine-level binary representation of a floating-point number (like binary64) into a human-readable decimal representation (like numbers in JSON), and back, since there exist published algorithms to do this exactly and optimally.<sup id="cite_ref-26" class="reference"><a href="#cite_note-26">&#91;26&#93;</a></sup>\n</p><p>Comments were purposefully excluded from JSON. In 2012, Douglas Crockford described his design decision thus: "I removed comments from JSON because I saw people were using them to hold parsing directives, a practice which would have destroyed interoperability."<sup id="cite_ref-DouglasCrockfordComments_25-1" class="reference"><a href="#cite_note-DouglasCrockfordComments-25">&#91;25&#93;</a></sup>\n</p><p>JSON disallows "trailing commas", a <a href="/wiki/Comma#Computing" title="Comma">comma</a> after the last value inside a data structure.<sup id="cite_ref-27" class="reference"><a href="#cite_note-27">&#91;27&#93;</a></sup> Trailing commas are a common feature of <a href="#Derivatives">JSON derivatives</a> to improve ease of use.<sup id="cite_ref-json5_28-0" class="reference"><a href="#cite_note-json5-28">&#91;28&#93;</a></sup>\n</p>\n<h3><span class="mw-headline" id="Semantics">Semantics</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JSON&amp;action=edit&amp;section=7" title="Edit section: Semantics">edit</a><span class="mw-editsection-bracket">]</span></span></h3>\n<p>While JSON provides a syntactic framework for data interchange, unambiguous data interchange also requires agreement between producer and consumer on the semantics of specific use of the JSON syntax.<sup id="cite_ref-29" class="reference"><a href="#cite_note-29">&#91;29&#93;</a></sup> One example of where such an agreement is necessary is the serialization of data types defined by the <a href="/wiki/JavaScript_syntax" title="JavaScript syntax">JavaScript syntax</a> that are not part of the JSON standard, e.g. Date, Function, Regular Expression, and <code>undefined</code>.<sup id="cite_ref-30" class="reference"><a href="#cite_note-30">&#91;30&#93;</a></sup>\n</p>\n<h2><span class="mw-headline" id="Metadata_and_schema">Metadata and schema</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JSON&amp;action=edit&amp;section=8" title="Edit section: Metadata and schema">edit</a><span class="mw-editsection-bracket">]</span></span></h2>\n<p>The official <a href="/wiki/MIME_type" class="mw-redirect" title="MIME type">MIME type</a> for JSON text is "<code>application/json</code>",<sup id="cite_ref-31" class="reference"><a href="#cite_note-31">&#91;31&#93;</a></sup> and most modern implementations have adopted this. The unofficial MIME type "<code>text/json</code>" or the content-type "<code>text/javascript</code>" are also supported for legacy reasons by many service providers, browsers, servers, web applications, libraries, frameworks, and APIs. Notable examples include the Google Search API,<sup id="cite_ref-faraday-stack-pull-2_32-0" class="reference"><a href="#cite_note-faraday-stack-pull-2-32">&#91;32&#93;</a></sup> Yahoo!,<sup id="cite_ref-faraday-stack-pull-2_32-1" class="reference"><a href="#cite_note-faraday-stack-pull-2-32">&#91;32&#93;</a></sup><sup id="cite_ref-33" class="reference"><a href="#cite_note-33">&#91;33&#93;</a></sup> Flickr,<sup id="cite_ref-faraday-stack-pull-2_32-2" class="reference"><a href="#cite_note-faraday-stack-pull-2-32">&#91;32&#93;</a></sup> Facebook API,<sup id="cite_ref-34" class="reference"><a href="#cite_note-34">&#91;34&#93;</a></sup> <a href="/wiki/Lift_(web_framework)" title="Lift (web framework)">Lift framework</a>,<sup id="cite_ref-35" class="reference"><a href="#cite_note-35">&#91;35&#93;</a></sup> Dojo Toolkit 0.4,<sup id="cite_ref-36" class="reference"><a href="#cite_note-36">&#91;36&#93;</a></sup> etc.\n</p><p><i>JSON Schema</i> specifies a JSON-based format to define the structure of JSON data for validation, documentation, and interaction control. It provides a contract for the JSON data required by a given application, and how that data can be modified.<sup id="cite_ref-37" class="reference"><a href="#cite_note-37">&#91;37&#93;</a></sup> JSON Schema is based on the concepts from <a href="/wiki/XML_Schema_(W3C)" title="XML Schema (W3C)">XML Schema</a> (XSD), but is JSON-based. As in XSD, the same serialization/deserialization tools can be used both for the schema and data, and it is self-describing. It is specified in an <a href="/wiki/Internet_Draft" title="Internet Draft">Internet Draft</a> at the IETF, currently in 2019-09 draft, which was released on September 19, 2019.<sup id="cite_ref-38" class="reference"><a href="#cite_note-38">&#91;38&#93;</a></sup> There are several validators available for different programming languages,<sup id="cite_ref-39" class="reference"><a href="#cite_note-39">&#91;39&#93;</a></sup> each with varying levels of conformance. There is no standard filename extension, but some have suggested <code>.schema.json</code>.<sup id="cite_ref-40" class="reference"><a href="#cite_note-40">&#91;40&#93;</a></sup>\n</p><p>The JSON standard does not support object <a href="/wiki/Reference_(computer_science)" title="Reference (computer science)">references</a>, but an <a href="/wiki/IETF" class="mw-redirect" title="IETF">IETF</a> draft standard for JSON-based object references exists.<sup id="cite_ref-41" class="reference"><a href="#cite_note-41">&#91;41&#93;</a></sup> The <a href="/wiki/Dojo_Toolkit" title="Dojo Toolkit">Dojo Toolkit</a> supports object references using standard JSON; specifically, the <code>dojox.json.ref</code> module provides support for several forms of referencing including <a href="/wiki/Circular_reference" title="Circular reference">circular</a>, multiple, inter-message, and <a href="/wiki/Lazy_evaluation" title="Lazy evaluation">lazy</a> referencing. Internally both do so by assigning a <code class="mw-highlight mw-highlight-lang-text mw-content-ltr" id="" style="" dir="ltr">&quot;$ref&quot;</code> key for such references and resolving it at parse-time; the IETF draft only specifies the URL syntax, but Dojo allows more.<sup id="cite_ref-42" class="reference"><a href="#cite_note-42">&#91;42&#93;</a></sup><sup id="cite_ref-43" class="reference"><a href="#cite_note-43">&#91;43&#93;</a></sup><sup id="cite_ref-44" class="reference"><a href="#cite_note-44">&#91;44&#93;</a></sup> Alternatively, non-standard solutions exist such as the use of Mozilla JavaScript Sharp Variables. However this functionality became obsolete with JavaScript 1.8.5 and was removed in Firefox version 12.<sup id="cite_ref-45" class="reference"><a href="#cite_note-45">&#91;45&#93;</a></sup>\n</p>\n<h2><span class="mw-headline" id="Uses">Uses</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JSON&amp;action=edit&amp;section=9" title="Edit section: Uses">edit</a><span class="mw-editsection-bracket">]</span></span></h2>\n<p><a href="/wiki/JSON-RPC" title="JSON-RPC">JSON-RPC</a> is a <a href="/wiki/Remote_procedure_call" title="Remote procedure call">remote procedure call</a> (RPC) protocol built on JSON, as a replacement for <a href="/wiki/XML-RPC" title="XML-RPC">XML-RPC</a> or <a href="/wiki/SOAP" title="SOAP">SOAP</a>. It is a simple protocol that defines only a handful of data types and commands. JSON-RPC lets a system send notifications (information to the server that does not require a response) and multiple calls to the server that can be answered out of order.\n</p><p><a href="/wiki/Asynchronous_JavaScript_and_JSON" class="mw-redirect" title="Asynchronous JavaScript and JSON">Asynchronous JavaScript and JSON</a> (or AJAJ) refers to the same <a href="/wiki/Dynamic_web_page" title="Dynamic web page">dynamic web page</a> methodology as <a href="/wiki/Ajax_(programming)" title="Ajax (programming)">Ajax</a>, but instead of <a href="/wiki/XML" title="XML">XML</a>, JSON is the data format. AJAJ is a web development technique that provides for the ability of a <a href="/wiki/Webpage" class="mw-redirect" title="Webpage">webpage</a> to request new data after it has loaded into the <a href="/wiki/Web_browser" title="Web browser">web browser</a>. Typically it renders new data from the server in response to user actions on that webpage. For example, what the user types into a <a href="/wiki/Text_box" title="Text box">search box</a>, <a href="/wiki/Client-side_scripting" class="mw-redirect" title="Client-side scripting">client-side code</a> then sends to the server, which immediately responds with a <a href="/wiki/Drop-down_list" title="Drop-down list">drop-down list</a> of matching <a href="/wiki/Database" title="Database">database</a> items.\n</p><p>While JSON is a data serialization format, it has seen <a href="/wiki/Ad_hoc" title="Ad hoc">ad hoc</a> usage as a <a href="/wiki/Configuration_file" title="Configuration file">configuration language</a>. In this use case, support for <a href="/wiki/Comment_(computer_programming)" title="Comment (computer programming)">comments</a> and other features have been deemed useful, which has led to several nonstandard JSON <a href="/wiki/Superset" class="mw-redirect" title="Superset">supersets</a> being created. Among them are HJSON,<sup id="cite_ref-46" class="reference"><a href="#cite_note-46">&#91;46&#93;</a></sup> <a href="/wiki/HOCON" title="HOCON">HOCON</a>, and JSON5 (which despite its name, isn\'t the fifth version of JSON).<sup id="cite_ref-lucidchart_47-0" class="reference"><a href="#cite_note-lucidchart-47">&#91;47&#93;</a></sup><sup id="cite_ref-48" class="reference"><a href="#cite_note-48">&#91;48&#93;</a></sup> The primary objective of version 1.2 of <a href="/wiki/YAML" title="YAML">YAML</a> was to make the nonstandard format a strict JSON superset.<sup id="cite_ref-YAML_Version_1.2_49-0" class="reference"><a href="#cite_note-YAML_Version_1.2-49">&#91;49&#93;</a></sup>\n</p><p>In 2012, Douglas Crockford had this to say about comments in JSON when used as a configuration language: "I know that the lack of comments makes some people sad, but it shouldn\'t. Suppose you are using JSON to keep configuration files, which you would like to annotate. Go ahead and insert all the comments you like. Then pipe it through JSMin<sup id="cite_ref-50" class="reference"><a href="#cite_note-50">&#91;50&#93;</a></sup> before handing it to your JSON parser."<sup id="cite_ref-DouglasCrockfordComments_25-2" class="reference"><a href="#cite_note-DouglasCrockfordComments-25">&#91;25&#93;</a></sup>\n</p><p>JSON is intended as a <a href="/wiki/Data_serialization" class="mw-redirect" title="Data serialization">data serialization</a> format. However, its design as a subset of JavaScript can lead to the misconception that it is safe to pass JSON texts to the JavaScript <code class="mw-highlight mw-highlight-lang-javascript mw-content-ltr" id="" style="" dir="ltr"><span class="nb">eval</span><span class="p">()</span></code> function. This is not safe, due to certain valid JSON texts, specifically those containing <span class="nowrap"><link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r886049734"/><span class="monospaced">U+2028</span>&#x20;</span> <span style="font-size:85%;">LINE SEPARATOR</span> or <span class="nowrap"><link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r886049734"/><span class="monospaced">U+2029</span>&#x20;</span> <span style="font-size:85%;">PARAGRAPH SEPARATOR</span>, not being valid JavaScript code until JavaScript specifications were updated in 2019, and so older engines may not support it.<sup id="cite_ref-Magnus_Holm_51-0" class="reference"><a href="#cite_note-Magnus_Holm-51">&#91;51&#93;</a></sup> To avoid the many pitfalls caused by executing arbitrary code from the Internet, a new function, <code class="mw-highlight mw-highlight-lang-javascript mw-content-ltr" id="" style="" dir="ltr"><span class="nx">JSON</span><span class="p">.</span><span class="nx">parse</span><span class="p">()</span></code> was first added to the fifth edition of ECMAScript,<sup id="cite_ref-52" class="reference"><a href="#cite_note-52">&#91;52&#93;</a></sup> which as of 2017 is supported by all major browsers. For non-supported browsers, an API-compatible JavaScript library is provided by <a href="/wiki/Douglas_Crockford" title="Douglas Crockford">Douglas Crockford</a>.<sup id="cite_ref-53" class="reference"><a href="#cite_note-53">&#91;53&#93;</a></sup> In addition, the TC39 proposal <a rel="nofollow" class="external text" href="https://github.com/tc39/proposal-json-superset">"Subsume JSON"</a> made <a href="/wiki/ECMAScript" title="ECMAScript">ECMAScript</a> a strict JSON superset as of the language\'s 2019 revision.<sup id="cite_ref-ECMATC39_22-1" class="reference"><a href="#cite_note-ECMATC39-22">&#91;22&#93;</a></sup><sup id="cite_ref-ECMATC39Stage4_23-1" class="reference"><a href="#cite_note-ECMATC39Stage4-23">&#91;23&#93;</a></sup>\n</p><p>Various JSON parser implementations have suffered from <a href="/wiki/Denial-of-service_attack" title="Denial-of-service attack">denial-of-service attack</a> and <a href="/wiki/Mass_assignment_vulnerability" title="Mass assignment vulnerability">mass assignment vulnerability</a>.<sup id="cite_ref-54" class="reference"><a href="#cite_note-54">&#91;54&#93;</a></sup><sup id="cite_ref-55" class="reference"><a href="#cite_note-55">&#91;55&#93;</a></sup>\n</p>\n<h2><span class="mw-headline" id="Comparison_with_other_formats">Comparison with other formats</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JSON&amp;action=edit&amp;section=10" title="Edit section: Comparison with other formats">edit</a><span class="mw-editsection-bracket">]</span></span></h2>\n<div role="note" class="hatnote navigation-not-searchable">See also: <a href="/wiki/Comparison_of_data_serialization_formats" class="mw-redirect" title="Comparison of data serialization formats">Comparison of data serialization formats</a></div>\n<p>JSON is promoted as a low-overhead alternative to XML as both of these formats have widespread support for creation, reading, and decoding in the real-world situations where they are commonly used.<sup id="cite_ref-56" class="reference"><a href="#cite_note-56">&#91;56&#93;</a></sup> Apart from XML, examples could include <a href="/wiki/Comma-separated_values" title="Comma-separated values">CSV</a> and <a href="/wiki/YAML" title="YAML">YAML</a> (a superset of JSON). Also, <a href="/wiki/Protocol_Buffers" title="Protocol Buffers">Google Protocol Buffers</a> can fill this role, although it is not a data interchange language.\n</p>\n<h3><span class="mw-headline" id="YAML">YAML</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JSON&amp;action=edit&amp;section=11" title="Edit section: YAML">edit</a><span class="mw-editsection-bracket">]</span></span></h3>\n<p><a href="/wiki/YAML" title="YAML">YAML</a> version 1.2 is a superset of JSON; prior versions were not strictly compatible. For example, escaping a slash <code class="mw-highlight mw-highlight-lang-text mw-content-ltr" id="" style="" dir="ltr">/</code> with a backslash <code class="mw-highlight mw-highlight-lang-text mw-content-ltr" id="" style="" dir="ltr">\\</code> is valid in JSON, but was not valid in YAML.<sup id="cite_ref-YAML_Version_1.2_49-1" class="reference"><a href="#cite_note-YAML_Version_1.2-49">&#91;49&#93;</a></sup> YAML supports comments, while JSON does not.<sup id="cite_ref-YAML_Version_1.2_49-2" class="reference"><a href="#cite_note-YAML_Version_1.2-49">&#91;49&#93;</a></sup><sup id="cite_ref-lucidchart_47-1" class="reference"><a href="#cite_note-lucidchart-47">&#91;47&#93;</a></sup><sup id="cite_ref-DouglasCrockfordComments_25-3" class="reference"><a href="#cite_note-DouglasCrockfordComments-25">&#91;25&#93;</a></sup>\n</p>\n<h3><span class="mw-headline" id="XML">XML</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JSON&amp;action=edit&amp;section=12" title="Edit section: XML">edit</a><span class="mw-editsection-bracket">]</span></span></h3>\n<p><a href="/wiki/XML" title="XML">XML</a> has been used to describe structured data and to serialize objects. Various XML-based protocols exist to represent the same kind of data structures as JSON for the same kind of data interchange purposes. Data can be encoded in XML in several ways. The most expansive form using tag pairs results in a much larger representation than JSON, but if data is stored in attributes and \'short tag\' form where the closing tag is replaced with <code class="mw-highlight mw-highlight-lang-text mw-content-ltr" id="" style="" dir="ltr">/&gt;</code>, the representation is often about the same size as JSON or just a little larger. However, an XML attribute can only have a single value and each attribute can appear at most once on each element.\n</p><p>XML separates "data" from "metadata" (via the use of elements and attributes), while JSON does not have such a concept.\n</p><p>Another key difference is the addressing of values. JSON has objects with a simple "key" to "value" mapping, whereas in XML addressing happens on "nodes", which all receive a unique ID via the XML processor. Additionally, the XML standard defines a common attribute <code class="mw-highlight mw-highlight-lang-text mw-content-ltr" id="" style="" dir="ltr">xml:id</code>, that can be used by the user, to set an ID explicitly.\n</p><p>XML tag names cannot contain any of the characters <code>!"#$%&amp;\'()*+,/;&lt;=&gt;?@[\\]^`{|}~</code>, nor a space character, and cannot begin with <code class="mw-highlight mw-highlight-lang-text mw-content-ltr" id="" style="" dir="ltr">-</code>, <code class="mw-highlight mw-highlight-lang-text mw-content-ltr" id="" style="" dir="ltr">.</code>, or a numeric digit, whereas JSON keys can (even if quotation mark and backslash must be escaped).<sup id="cite_ref-XML_1.1_57-0" class="reference"><a href="#cite_note-XML_1.1-57">&#91;57&#93;</a></sup>\n</p><p>XML values are strings of <i>characters</i>, with no built-in <a href="/wiki/Type_safety" title="Type safety">type safety</a>. XML has the concept of <a href="/wiki/XML_schema" title="XML schema">schema</a>, that permits strong typing, user-defined types, predefined tags, and formal structure, allowing for formal validation of an XML stream. JSON has strong typing built-in, and has a similar schema concept in <a href="#JSON_Schema">JSON Schema</a>.\n</p><p>XML supports comments, while JSON does not.<sup id="cite_ref-Saternos45_58-0" class="reference"><a href="#cite_note-Saternos45-58">&#91;58&#93;</a></sup><sup id="cite_ref-DouglasCrockfordComments_25-4" class="reference"><a href="#cite_note-DouglasCrockfordComments-25">&#91;25&#93;</a></sup>\n</p>\n<h2><span class="mw-headline" id="Derivatives">Derivatives</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JSON&amp;action=edit&amp;section=13" title="Edit section: Derivatives">edit</a><span class="mw-editsection-bracket">]</span></span></h2>\n<table class="box-More_citations_needed plainlinks metadata ambox ambox-content ambox-Refimprove" role="presentation"><tbody><tr><td class="mbox-image"><div style="width:52px"><a href="/wiki/File:Question_book-new.svg" class="image"><img alt="" src="//upload.wikimedia.org/wikipedia/en/thumb/9/99/Question_book-new.svg/50px-Question_book-new.svg.png" decoding="async" width="50" height="39" srcset="//upload.wikimedia.org/wikipedia/en/thumb/9/99/Question_book-new.svg/75px-Question_book-new.svg.png 1.5x, //upload.wikimedia.org/wikipedia/en/thumb/9/99/Question_book-new.svg/100px-Question_book-new.svg.png 2x" data-file-width="512" data-file-height="399" /></a></div></td><td class="mbox-text"><div class="mbox-text-span">This section <b>needs additional citations for <a href="/wiki/Wikipedia:Verifiability" title="Wikipedia:Verifiability">verification</a></b>.<span class="hide-when-compact"> Please help <a class="external text" href="https://en.wikipedia.org/w/index.php?title=JSON&amp;action=edit">improve this article</a> by <a href="/wiki/Help:Referencing_for_beginners" title="Help:Referencing for beginners">adding citations to reliable sources</a>.
