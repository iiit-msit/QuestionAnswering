search</a>\n\t\t<div id="mw-content-text" lang="en" dir="ltr" class="mw-content-ltr"><div class="mw-parser-output"><div class="shortdescription nomobile noexcerpt noprint searchaux" style="display:none">The creation of knowledge from structured and unstructured sources</div>\n<p><b>Knowledge extraction</b> is the creation of <a href="/wiki/Knowledge_representation_and_reasoning" title="Knowledge representation and reasoning">knowledge</a> from structured (<a href="/wiki/Relational_databases" class="mw-redirect" title="Relational databases">relational databases</a>, <a href="/wiki/XML" title="XML">XML</a>) and unstructured (<a href="/wiki/Text_(literary_theory)" title="Text (literary theory)">text</a>, documents, <a href="/wiki/Image" title="Image">images</a>) sources. The resulting knowledge needs to be in a machine-readable and machine-interpretable format and must <a href="/wiki/Knowledge_representation_and_reasoning" title="Knowledge representation and reasoning">represent knowledge</a> in a manner that facilitates inferencing. Although it is methodically similar to <a href="/wiki/Information_extraction" title="Information extraction">information extraction</a> (<a href="/wiki/Natural_language_processing" title="Natural language processing">NLP</a>) and <a href="/wiki/Extract,_transform,_load" title="Extract, transform, load">ETL</a> (data warehouse), the main criteria is that the extraction result goes beyond the creation of structured information or the transformation into a <a href="/wiki/Database_schema" title="Database schema">relational schema</a>. It requires either the reuse of existing <a href="/wiki/Knowledge_representation_and_reasoning" title="Knowledge representation and reasoning">formal knowledge</a> (reusing identifiers or <a href="/wiki/Ontologies" class="mw-redirect" title="Ontologies">ontologies</a>) or the generation of a schema based on the source data.\n</p><p>The RDB2RDF W3C group <sup id="cite_ref-RDB2RDF_1-0" class="reference"><a href="#cite_note-RDB2RDF-1">&#91;1&#93;</a></sup> is currently standardizing a language for extraction of <a href="/wiki/Resource_Description_Framework" title="Resource Description Framework">resource description frameworks</a> (RDF) from <a href="/wiki/Relational_databases" class="mw-redirect" title="Relational databases">relational databases</a>. Another popular example for knowledge extraction is the transformation of Wikipedia into <a href="/wiki/Structured_data" class="mw-redirect" title="Structured data">structured data</a> and also the mapping to existing <a href="/wiki/Knowledge_representation_and_reasoning" title="Knowledge representation and reasoning">knowledge</a> (see <a href="/wiki/DBpedia" title="DBpedia">DBpedia</a> and <a href="/wiki/Freebase_(database)" title="Freebase (database)">Freebase</a>).\n</p>\n<div id="toc" class="toc" role="navigation" aria-labelledby="mw-toc-heading"><input type="checkbox" role="button" class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Knowledge_extraction&amp;action=edit&amp;section=1" title="Edit section: Overview">edit</a><span class="mw-editsection-bracket">]</span></span></h2>\n<p>After the standardization of knowledge representation languages such as <a href="/wiki/Resource_Description_Framework" title="Resource Description Framework">RDF</a> and <a href="/wiki/Web_Ontology_Language" title="Web Ontology Language">OWL</a>, much research has been conducted in the area, especially regarding transforming relational databases into RDF, <a href="/wiki/Identity_resolution" class="mw-redirect" title="Identity resolution">identity resolution</a>, <a href="/wiki/Knowledge_discovery" class="mw-redirect" title="Knowledge discovery">knowledge discovery</a> and ontology learning. The general process uses traditional methods from <a href="/wiki/Information_extraction" title="Information extraction">information extraction</a> and <a href="/wiki/Extract,_transform,_load" title="Extract, transform, load">extract, transform, and load</a> (ETL), which transform the data from the sources into structured formats.\n</p><p>The following criteria can be used to categorize approaches in this topic (some of them only account for extraction from relational databases):<sup id="cite_ref-lod2_eu_2-0" class="reference"><a href="#cite_note-lod2_eu-2">&#91;2&#93;</a></sup>\n</p>\n<table class="wikitable">\n\n<tbody><tr>\n<th>Source\n</th>\n<td>Which data sources are covered: Text, Relational Databases, XML, CSV\n</td></tr>\n<tr>\n<th>Exposition\n</th>\n<td>How is the extracted knowledge made explicit (ontology file, semantic database)? How can you query it?\n</td></tr>\n<tr>\n<th>Synchronization\n</th>\n<td>Is the knowledge extraction process executed once to produce a dump or is the result synchronized with the source? Static or dynamic. Are changes to the result written back (bi-directional)\n</td></tr>\n<tr>\n<th>Reuse of vocabularies\n</th>\n<td>The tool is able to reuse existing vocabularies in the extraction. For example, the table column \'firstName\' can be mapped to foaf:firstName. Some automatic approaches are not capable of mapping vocab.\n</td></tr>\n<tr>\n<th>Automatization\n</th>\n<td>The degree to which the extraction is assisted/automated. Manual, GUI, semi-automatic, automatic.\n</td></tr>\n<tr>\n<th>Requires a domain ontology\n</th>\n<td>A pre-existing ontology is needed to map to it. So either a mapping is created or a schema is learned from the source (<a href="/wiki/Ontology_learning" title="Ontology learning">ontology learning</a>).\n</td></tr></tbody></table>\n<h2><span Counter examples: Methods that only recognize entities or link to Wikipedia articles and other targets that do not provide further retrieval of structured data and formal knowledge.</dd></dl>\n<h3><span class="mw-headline" id="Relational_databases_to_RDF">Relational databases to RDF</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Knowledge_extraction&amp;action=edit&amp;section=4" title="Edit section: Relational databases to RDF">edit</a><span class="mw-editsection-bracket">]</span></span></h3>\n<ol><li><a href="/w/index.php?title=Triplify&amp;action=edit&amp;redlink=1" class="new" title="Triplify (page does not exist)">Triplify</a>, D2R Server, <a rel="nofollow" class="external text" href="https://capsenta.com/#section-ultrawrap">Ultrawrap</a>, and <a href="/wiki/Virtuoso_Universal_Server" title="Virtuoso Universal Server">Virtuoso</a> RDF Views are tools that transform relational databases to RDF. During this process they allow reusing existing vocabularies and <a href="/wiki/Ontology_(information_science)" title="Ontology (information science)">ontologies</a> during the conversion process. When transforming a typical relational table named <i>users</i>, one column (e.g.<i>name</i>) or an aggregation of columns (e.g.<i>first_name</i> and <i>last_name</i>) has to provide the URI of the created entity. Normally the primary key is used. Every other column can be extracted as a relation with this entity.<sup id="cite_ref-timbl_reldb4semweb_4-0" class="reference"><a href="#cite_note-timbl_reldb4semweb-4">&#91;4&#93;</a></sup> Then properties with formally defined semantics are used (and reused) to interpret the information. For example, a column in a user table called <i>marriedTo</i> can be defined as symmetrical relation and a column <i>homepage</i> can be converted to a property from the <a href="/wiki/FOAF_(software)" class="mw-redirect" title="FOAF (software)">FOAF Vocabulary</a> called <a rel="nofollow" class="external text" href="http://xmlns.com/foaf/spec/#term_homepage">foaf:homepage</a>, thus qualifying it as an <a href="/wiki/Web_Ontology_Language#Properties" title="Web Ontology Language">inverse functional property</a>. Then each entry of the <i>user</i> table can be made an instance of the class <a rel="nofollow" class="external text" href="http://xmlns.com/foaf/spec/#term_Person">foaf:Person</a> (Ontology Population). Additionally <a href="/wiki/Domain_knowledge" title="Domain knowledge">domain knowledge</a> (in form of an ontology) could be created from the <i>status_id</i>, either by manually created rules (if <i>status_id</i> is 2, the entry belongs to class Teacher ) or by (semi)-automated methods (<a href="/wiki/Ontology_learning" title="Ontology learning">ontology learning</a>). class="mw-editsection-bracket">]</span></span></h2>\n<h3><span id="1:1_Mapping_from_RDB_Tables.2FViews_to_RDF_Entities.2FAttributes.2FValues"></span><span class="mw-headline" id="1:1_Mapping_from_RDB_Tables/Views_to_RDF_Entities/Attributes/Values">1:1 Mapping from RDB Tables/Views to RDF Entities/Attributes/Values</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Knowledge_extraction&amp;action=edit&amp;section=6" title="Edit section: 1:1 Mapping from RDB Tables/Views to RDF Entities/Attributes/Values">edit</a><span class="mw-editsection-bracket">]</span></span></h3>\n<p>When building a RDB representation of a problem domain, the starting point is frequently an entity-relationship diagram (ERD). Typically, each entity is represented as a database table, each attribute of the entity becomes a column in that table, and relationships between entities are indicated by foreign keys. Each table typically defines a particular class of entity, each column one of its attributes. Each row in the table describes an entity\ninstance, uniquely identified by a primary key. The table rows collectively describe an entity set. In an equivalent RDF representation of the same entity set:\n</p>\n<ul><li>Each column in the table is an attribute (i.e., predicate)</li>\n<li>Each column value is an attribute value (i.e., object)</li>\n<li>Each row key represents an entity ID (i.e., subject)</li>\n<li>Each row represents an entity instance</li>\n<li>Each row (entity instance) is represented in RDF by a collection of triples with a common subject (entity ID).</li></ul>\n<p>So, to render an equivalent view based on RDF semantics, the basic mapping algorithm would be as follows:\n</p>\n<ol><li>create an RDFS class for each table</li>\n<li>convert all primary keys and foreign keys into IRIs</li>\n<li>assign a predicate IRI to each column</li>\n<li>assign an rdf:type predicate for each row, linking it to an RDFS class IRI corresponding to the table</li>\n<li>for each column that is neither part of a primary or foreign key, construct a triple containing the primary key IRI as the subject, the column IRI as the predicate and the column\'s value as the object.</li></ol>\n<p>Early mentioning of this basic or direct mapping can be found in <a href="/wiki/Tim_Berners-Lee" title="Tim Berners-Lee">Tim Berners-Lee</a>\'s comparison of the <a href="/wiki/Entity-relationship_model" class="mw-redirect" title="Entity-relationship model">ER model</a> to the RDF model.<sup id="cite_ref-timbl_reldb4semweb_4-1" class="reference"><a href="#cite_note-timbl_reldb4semweb-4">&#91;4&#93;</a></sup>\n</p>\n<h3><span class="mw-headline" id="Complex_mappings_of_relational_databases_to_RDF">Complex mappings of relational databases to RDF</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Knowledge_extraction&amp;action=edit&amp;section=7" title="Edit section: Complex mappings of relational databases to RDF">edit</a><span class="mw-editsection-bracket">]</span></span></h3>\n<p>The 1:1 mapping mentioned above exposes the legacy data as RDF in a straightforward way, additional refinements can be employed to improve the usefulness of RDF output respective the given Use Cases. Normally, information is lost during the transformation of an entity-relationship diagram (ERD) to relational tables (Details can be found in <a href="/wiki/Object-relational_impedance_mismatch" class="mw-redirect" title="Object-relational impedance mismatch">object-relational impedance mismatch</a>) and has to be <a href="/wiki/Reverse_Engineering" class="mw-redirect" title="Reverse Engineering">reverse engineered</a>. From a conceptual view, approaches for extraction can come from two directions. The first direction tries to extract or learn an OWL schema from the given database schema. Early approaches used a fixed amount of manually created mapping rules to refine the 1:1 mapping.<sup id="cite_ref-Hu_5-0" class="reference"><a href="#cite_note-Hu-5">&#91;5&#93;</a></sup><sup id="cite_ref-Ghawi_6-0" class="reference"><a href="#cite_note-Ghawi-6">&#91;6&#93;</a></sup><sup id="cite_ref-Li_7-0" class="reference"><a href="#cite_note-Li-7">&#91;7&#93;</a></sup> More elaborate methods are employing heuristics or learning algorithms to induce schematic information (methods overlap with <a href="/wiki/Ontology_learning" title="Ontology learning">ontology learning</a>). While some approaches try to extract the information from the structure inherent in the SQL schema<sup id="cite_ref-Tirmizi_8-0" class="reference"><a href="#cite_note-Tirmizi-8">&#91;8&#93;</a></sup> (analysing e.g. foreign keys), others analyse the content and the values in the tables to create conceptual hierarchies<sup id="cite_ref-Cerbah_9-0" class="reference"><a href="#cite_note-Cerbah-9">&#91;9&#93;</a></sup> (e.g. a columns with few values are candidates for becoming categories). The second direction tries to map the schema and its contents to a pre-existing domain ontology (see also: <a href="/wiki/Ontology_alignment" title="Ontology alignment">ontology alignment</a>). Often, however, a suitable domain ontology does not exist and has to be created first.\n</p>\n<h3><span class="mw-headline" id="XML">XML</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Knowledge_extraction&amp;action=edit&amp;section=8" title="Edit section: XML">edit</a><span class="mw-editsection-bracket">]</span></span></h3>\n<p>As XML is structured as a tree, any data can be easily represented in RDF, which is structured as a graph. <a rel="nofollow" class="external text" href="http://rhizomik.net/html/redefer/xml2rdf/">XML2RDF</a> is one example of an approach that uses RDF blank nodes and transforms XML elements and attributes to RDF properties. The topic however is more complex as in the case of relational databases. In a relational table the primary key is an ideal candidate for becoming the subject of the extracted triples. An XML element, however, can be transformed - depending on the context- as a subject, a predicate or object of a triple. <a href="/wiki/XSLT" title="XSLT">XSLT</a> can be used a standard transformation language to manually convert XML to RDF.\n</p>\n<h3><span id="Survey_of_methods_.2F_tools"></span><span class="mw-headline" id="Survey_of_methods_/_tools">Survey of methods / tools</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Knowledge_extraction&amp;action=edit&amp;section=9" title="Edit section: Survey of methods / tools">edit</a><span class="mw-editsection-bracket">]</span></span></h3>\n<table class="wikitable sortable">\n\n<tbody><tr>\n<th>Name</th>\n<th>Data Source</th>\n<th>Data Exposition</th>\n<th>Data Synchronisation</th>\n<th>Mapping Language</th>\n<th>Vocabulary Reuse</th>\n<th>Mapping Automat.</th>\n<th>Req. id="Extraction_from_natural_language_sources">Extraction from natural language sources</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Knowledge_extraction&amp;action=edit&amp;section=10" title="Edit section: Extraction from natural language sources">edit</a><span class="mw-editsection-bracket">]</span></span></h2>\n<p>The largest portion of information contained in business documents (about 80%<sup id="cite_ref-Wimalasuriya_10-0" class="reference"><a href="#cite_note-Wimalasuriya-10">&#91;10&#93;</a></sup>) is encoded in natural language and therefore unstructured. Because <a href="/wiki/Unstructured_data" title="Unstructured data">unstructured data</a> is rather a challenge for knowledge extraction, more sophisticated methods are required, which generally tend to supply worse results compared to structured data. The potential for a massive acquisition of extracted knowledge, however, should compensate the increased complexity and decreased quality of extraction. In the following, natural language sources are understood as sources of information, where the data is given in an unstructured fashion as plain text.  If the given text is additionally embedded in a markup document (e. g. HTML document), the mentioned systems normally remove the markup elements automatically.\n</p>\n<h3><span id="Linguistic_annotation_.2F_natural_language_processing_.28NLP.29"></span><span class="mw-headline" id="Linguistic_annotation_/_natural_language_processing_(NLP)">Linguistic annotation / natural language processing (NLP)</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Knowledge_extraction&amp;action=edit&amp;section=11" title="Edit section: Linguistic annotation / natural language processing (NLP)">edit</a><span class="mw-editsection-bracket">]</span></span></h3>\n<div role="note" class="hatnote navigation-not-searchable">Main articles: <a href="/wiki/Natural_language_processing" title="Natural language processing">Natural language processing</a> and <a href="/wiki/Linguistic_Annotation" class="mw-redirect" title="Linguistic Annotation">Linguistic Annotation</a></div>\n<p>As a preprocessing step to knowledge extraction, it can be necessary to perform linguistic annotation by one or multiple <a href="/wiki/Natural_language_processing" title="Natural language processing">NLP</a> tools. Individual modules in an NLP workflow normally build on tool-specific formats for input and output, but in the context of knowledge extraction, structured formats for representing linguistic annotations have been applied.\n</p><p>Typical NLP tasks relevant to knowledge extraction include:\n</p>\n<ul><li>part-of-speech (POS) tagging</li>\n<li>lemmatization (LEMMA) or stemming (STEM)</li>\n<li>word sense disambiguation (WSD, related to semantic annotation below)</li>\n<li>named entity recognition (NER, also see IE below)</li>\n<li>syntactic parsing, often adopting syntactic dependencies (DEP)</li>\n<li>shallow syntactic parsing (CHUNK): if performance is an issue, chunking yields a fast extraction of nominal and other phrases</li>\n<li>anaphor resolution (see coreference resolution in IE below, but seen here as the task to create links between textual mentions rather than between the mention of an entity and an abstract representation of the entity)</li>\n<li>semantic role labelling (SRL, related to relation extraction; not to be confused with semantic annotation as described below)</li>\n<li>discourse parsing (relations between different sentences, rarely used in real-world applications)</li></ul>\n<p>In NLP, such data is typically represented in TSV formats (CSV formats with TAB as separators), often referred to as CoNLL formats. For knowledge extraction workflows, RDF views on such data have been created in accordance with the following community standards:\n</p>\n<ul><li>NLP Interchange Format (NIF, for many frequent types of annotation)<sup id="cite_ref-11" class="reference"><a href="#cite_note-11">&#91;11&#93;</a></sup><sup id="cite_ref-12" class="reference"><a href="#cite_note-12">&#91;12&#93;</a></sup></li>\n<li><a href="/wiki/Web_Annotation" class="mw-redirect" title="Web Annotation">Web Annotation</a> (WA, often used for entity linking)<sup id="cite_ref-13" class="reference"><a href="#cite_note-13">&#91;13&#93;</a></sup></li>\n<li>CoNLL-RDF (for annotations originally represented in TSV formats)<sup id="cite_ref-14" class="reference"><a href="#cite_note-14">&#91;14&#93;</a></sup><sup id="cite_ref-15" class="reference"><a href="#cite_note-15">&#91;15&#93;</a></sup></li></ul>\n<p>Other, platform-specific formats include\n</p>\n<ul><li>LAPPS Interchange Format (LIF, used in the LAPPS Grid)<sup id="cite_ref-16" class="reference"><a href="#cite_note-16">&#91;16&#93;</a></sup><sup id="cite_ref-17" class="reference"><a href="#cite_note-17">&#91;17&#93;</a></sup></li>\n<li>NLP Annotation Format (NAF, used in the NewsReader workflow management system)<sup id="cite_ref-18" class="reference"><a href="#cite_note-18">&#91;18&#93;</a></sup><sup id="cite_ref-19" class="reference"><a href="#cite_note-19">&#91;19&#93;</a></sup></li></ul>\n<h3><span id="Traditional_information_extraction_.28IE.29"></span><span class="mw-headline" id="Traditional_information_extraction_(IE)">Traditional information extraction (IE)</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Knowledge_extraction&amp;action=edit&amp;section=12" title="Edit section: Traditional information extraction (IE)">edit</a><span class="mw-editsection-bracket">]</span></span></h3>\n<p>Traditional <a href="/wiki/Information_extraction" title="Information extraction">information extraction</a> <sup id="cite_ref-Cunningham_20-0" class="reference"><a href="#cite_note-Cunningham-20">&#91;20&#93;</a></sup> is a technology of natural language processing, which extracts information from typically natural language texts and structures these in a suitable manner. The kinds of information to be identified must be specified in a model before beginning the process, which is why the whole process of traditional Information Extraction is domain dependent. The IE is split in the following five subtasks.\n</p>\n<ul><li><a href="/wiki/Named-entity_recognition" title="Named-entity recognition">Named entity recognition</a> (NER)</li>\n<li><a href="/wiki/Coreference" title="Coreference">Coreference resolution</a> (CO)</li>\n<li>Template element construction (TE)</li>\n<li>Template relation construction (TR)</li>\n<li>Template scenario production (ST)</li></ul>\n<p>The task of <a href="/wiki/Named_entity_recognition" class="mw-redirect" title="Named entity recognition">named entity recognition</a> is to recognize and to categorize all named entities contained in a text (assignment of a named entity to a predefined category). This works by application of grammar based methods or statistical models.\n</p><p>Coreference resolution identifies equivalent entities, which were recognized by NER, within a text. There are two relevant kinds of equivalence relationship. The first one relates to the relationship between two different represented entities (e.g. IBM Europe and IBM) and the second one to the relationship between an entity and their <a href="/wiki/Anaphora_(linguistics)" title="Anaphora (linguistics)">anaphoric references</a> (e.g. it and IBM). Both kinds can be recognized by coreference resolution.\n</p><p>During template element construction the IE system identifies descriptive properties of entities, recognized by NER and CO. These properties correspond to ordinary qualities like red or big.\n</p><p>Template relation construction identifies relations, which exist between the template elements. These relations can be of several kinds, such as works-for or located-in, with the restriction, that both domain and range correspond to entities.\n</p><p>In the template scenario production events, which are described in the text, will be identified and structured with respect to the entities, recognized by NER and CO and relations, identified by TR.\n</p>\n<h3><span id="Ontology-based_information_extraction_.28OBIE.29"></span><span class="mw-headline" id="Ontology-based_information_extraction_(OBIE)">Ontology-based information extraction (OBIE)</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Knowledge_extraction&amp;action=edit&amp;section=13" title="Edit section: Ontology-based information extraction (OBIE)">edit</a><span class="mw-editsection-bracket">]</span></span></h3>\n<p>Ontology-based information extraction <sup id="cite_ref-Wimalasuriya_10-1" class="reference"><a href="#cite_note-Wimalasuriya-10">&#91;10&#93;</a></sup> is a subfield of information extraction, with which at least one <a href="/wiki/Ontology_(information_science)" title="Ontology (information science)">ontology</a> is used to guide the process of information extraction from natural language text. The OBIE system uses methods of traditional information extraction to identify <a href="/wiki/Concept" title="Concept">concepts</a>, instances and relations of the used ontologies in the text, which will be structured to an ontology after the process. Thus, the input ontologies constitute the model of information to be extracted.\n</p>\n<h3><span id="Ontology_learning_.28OL.29"></span><span class="mw-headline" id="Ontology_learning_(OL)">Ontology learning (OL)</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Knowledge_extraction&amp;action=edit&amp;section=14" title="Edit section: Ontology learning (OL)">edit</a><span class="mw-editsection-bracket">]</span></span></h3>\n<div role="note" class="hatnote navigation-not-searchable">Main article: <a href="/wiki/Ontology_learning" title="Ontology learning">Ontology learning</a></div>\n<p>Ontology learning is the automatic or semi-automatic creation of ontologies, including extracting the corresponding domain\'s terms from natural language text. As building ontologies manually is extremely labor-intensive and time consuming, there is great motivation to automate the process.\n</p>\n<h3><span id="Semantic_annotation_.28SA.29"></span><span class="mw-headline" id="Semantic_annotation_(SA)">Semantic annotation (SA)</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Knowledge_extraction&amp;action=edit&amp;section=15" title="Edit section: Semantic annotation (SA)">edit</a><span class="mw-editsection-bracket">]</span></span></h3>\n<p>During semantic annotation,<sup id="cite_ref-Erdmann_21-0" class="reference"><a href="#cite_note-Erdmann-21">&#91;21&#93;</a></sup> natural language text is augmented with metadata (often represented in <a href="/wiki/RDFa" title="RDFa">RDFa</a>), which should make the semantics of contained terms machine-understandable. At this process, which is generally semi-automatic, knowledge is extracted in the sense, that a link between lexical terms and for example concepts from ontologies is established. Thus, knowledge is gained, which meaning of a term in the processed context was intended and therefore the meaning of the text is grounded in <a href="/wiki/Machine-readable_data" title="Machine-readable data">machine-readable data</a> with the ability to draw inferences. Semantic annotation is typically split into the following two subtasks.\n</p>\n<ol><li><a href="/wiki/Terminology_extraction" title="Terminology extraction">Terminology extraction</a></li>\n<li><a href="/wiki/Entity_linking" title="Entity linking">Entity linking</a></li></ol>\n<p>At the terminology extraction level, lexical terms from the text are extracted. For this purpose a tokenizer determines at first the word boundaries and solves abbreviations. Afterwards terms from the text, which correspond to a concept, are extracted with the help of a domain-specific lexicon to link these at entity linking.\n</p><p>In entity linking <sup id="cite_ref-Rao_22-0" class="reference"><a href="#cite_note-Rao-22">&#91;22&#93;</a></sup> a link between the extracted lexical terms from the source text and the concepts from an ontology or knowledge base such as <a href="/wiki/DBpedia" title="DBpedia">DBpedia</a>  is established. For this, candidate-concepts are detected appropriately to the several meanings of a term with the help of a lexicon. Finally, the context of the terms is analyzed to determine the most appropriate disambiguation and to assign the term to the correct concept.\n</p><p>Note that "semantic annotation" in the context of knowledge extraction is not to be confused with <a href="/wiki/Semantic_parsing" title="Semantic parsing">semantic parsing</a> as understood in natural language processing (also referred to as "semantic annotation"): Semantic parsing aims a complete, machine-readable representation of natural language, whereas semantic annotation in the sense of knowledge extraction tackles only a very elementary aspect of that.\n</p>\n<h3><span class="mw-headline" id="Tools">Tools</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Knowledge_extraction&amp;action=edit&amp;section=17" title="Edit section: Knowledge discovery">edit</a><span class="mw-editsection-bracket">]</span></span></h2>\n<p>Knowledge discovery describes the process of automatically searching large volumes of <a href="/wiki/Data" title="Data">data</a> for patterns that can be considered <a href="/wiki/Knowledge" title="Knowledge">knowledge</a> <i>about</i> the data.<sup id="cite_ref-Williams1992_43-0" class="reference"><a href="#cite_note-Williams1992-43">&#91;43&#93;</a></sup>  It is often described as <i>deriving</i> knowledge from the input data.  Knowledge discovery developed out of the <a href="/wiki/Data_mining" title="Data mining">data mining</a> domain, and is closely related to it both in terms of methodology and terminology.<sup id="cite_ref-Fayyad1996_44-0" class="reference"><a href="#cite_note-Fayyad1996-44">&#91;44&#93;</a></sup>\n</p><p>The most well-known branch of <a href="/wiki/Data_mining" title="Data mining">data mining</a> is knowledge discovery, also known as <a href="/wiki/Knowledge_discovery_in_databases" class="mw-redirect" title="Knowledge discovery in databases">knowledge discovery in databases</a> (KDD). Just as many other forms of knowledge discovery it creates <a href="/wiki/Abstraction" title="Abstraction">abstractions</a> of the input data. The <i>knowledge</i> obtained through the process may become additional <i>data</i> that can be used for further usage and discovery. Often the outcomes from knowledge discovery are not actionable, <a href="/wiki/Actionable_knowledge_discovery" class="mw-redirect" title="Actionable knowledge discovery">actionable knowledge discovery</a>, also known as <a href="/wiki/Domain_driven_data_mining" title="Domain driven data mining">domain driven data mining</a>,<sup id="cite_ref-45" class="reference"><a href="#cite_note-45">&#91;45&#93;</a></sup> aims to discover and deliver actionable knowledge and insights.\n</p><p>Another promising application of knowledge discovery is in the area of <a href="/wiki/Software_modernization" title="Software modernization">software modernization</a>, weakness discovery and compliance which involves understanding existing software artifacts. This process is related to a concept of <a href="/wiki/Reverse_engineering" title="Reverse engineering">reverse engineering</a>. Usually the knowledge obtained from existing software is presented in the form of models to which specific queries can be made when necessary. An <a href="/wiki/Entity_relationship" class="mw-redirect" title="Entity relationship">entity relationship</a> is a frequent format of representing knowledge obtained from existing software. <a href="/wiki/Object_Management_Group" title="Object Management Group">Object Management Group</a> (OMG) developed the specification <a href="/wiki/Knowledge_Discovery_Metamodel" title="Knowledge Discovery Metamodel">Knowledge Discovery Metamodel</a> (KDM) which defines an ontology for the software assets and their relationships for the purpose of performing knowledge discovery in existing code. Knowledge discovery from existing software systems, also known as <a href="/wiki/Software_mining" title="Software mining">software mining</a> is closely related to <a href="/wiki/Data_mining" title="Data mining">data mining</a>, since existing software artifacts contain enormous value for risk management and <a href="/wiki/Business_Value" class="mw-redirect" title="Business Value">business value</a>, key for the evaluation and evolution of software systems. Instead of mining individual <a href="/wiki/Data_set" title="Data set">data sets</a>, <a href="/wiki/Software_mining" title="Software mining">software mining</a> focuses on <a href="/wiki/Metadata" title="Metadata">metadata</a>, such as process flows (e.g. data flows, control flows, &amp; call maps), architecture, database schemas, and business rules/terms/process.\n</p>\n<h3><span
