<b>frame</b> is a part of a <a href="/wiki/Web_page" title="Web page">web page</a> or browser <a href="/wiki/Window_(computing)" title="Window (computing)">window</a> which displays content independent of its container, with the ability to load content independently. The <a href="/wiki/HTML" title="HTML">HTML</a> or media elements shown in a frame may come from a different <a href="/wiki/Website" title="Website">web site</a> as the other elements of content on display, although this practice, known as <b>framing</b>,<sup id="cite_ref-1" class="reference"><a href="#cite_note-1">&#91;1&#93;</a></sup> is today often regarded as a violation of <a href="/wiki/Same-origin_policy" title="Same-origin policy">same-origin policy</a> and has been considered a form of <a href="/wiki/Copyright_infringement" title="Copyright infringement">copyright infringement</a>.\n</p><p>In <a href="/wiki/HTML" title="HTML">HTML</a>, a <b>frameset</b> is a group of named frames to which web pages and media can be directed; an <b>iframe</b> provides for a frame to be placed inside the body of a document.\n</p><p>Since the early 2000s, the use of framesets has been considered obsolete due to usability and accessibility concerns, and the feature has been removed from the <a href="/wiki/HTML5" title="HTML5">HTML5</a> standard.\n</p>\n<div id="toc" class="toc" role="navigation" aria-labelledby="mw-toc-heading"><input type="checkbox" role="button" and Attributes">edit</a><span class="mw-editsection-bracket">]</span></span></h2>\n<p>The frames in HTML are created using the <code>&lt;frameset&gt;&lt;/frameset&gt;</code> tag pair. The <code>&lt;frameset&gt;</code> tag is a container tag for all other tags that are used to create frames. The <code>&lt;frameset&gt;</code> tag replaces the <code>&lt;body&gt;</code> tag in frameset documents. The <code>&lt;frameset&gt;</code> tag defines how to divide the window into frames. \n</p><p>Each frameset defines a set of rows or columns.&#160; If user define frames by using the <code>rows</code> attribute then horizontal frames are created. If user define frames by using <code>cols</code> then vertical frames are created.\n</p><p>The <code>&lt;noframes&gt;</code> element may be included so web browsers with frames disabled (or browsers If set to an integer greater than 0, the user can resize the frames by dragging this border, unless a <code>noresize</code> attribute is present in a frame element. If border is set to 0, no border will be displayed and content in different frames will abut each other without delineation.\n</p><p>The <code>iframe</code> element is used inline within a normal HTML body, and defines the initial content and name similarly to the <code>frame</code> element. 2008</span>)</i></small></div></td></tr></tbody></table>\n<p><a href="/wiki/Netscape_Navigator" title="Netscape Navigator">Netscape Navigator</a> 2.0 introduced the elements used for frames in March 1996. Other browser vendors such as <a href="/wiki/Apple_Inc." title="Apple Inc.">Apple</a> with <a href="/wiki/Cyberdog" title="Cyberdog">Cyberdog</a> followed later that year.<sup id="cite_ref-2" class="reference"><a href="#cite_note-2">&#91;2&#93;</a></sup> At that time, Netscape proposed frames to the <a href="/wiki/World_Wide_Web_Consortium" title="World Wide Web Consortium">World Wide Web Consortium</a> (W3C) for inclusion in the HTML 3.0 standard.<sup id="cite_ref-3" class="reference"><a href="#cite_note-3">&#91;3&#93;</a></sup>\n</p><p>Frames were used to display and navigate early <a href="/wiki/Online_magazine" title="Online magazine">online magazines</a> and <a href="/wiki/Web_apps" class="mw-redirect" title="Web apps">web apps</a>, such as <a href="/wiki/Webmail" title="Webmail">webmail</a> services and <a href="/wiki/Web_chat" title="Web chat">web chat</a> sites. Frames had the advantage of allowing elements to be displayed sitewide without requiring server features such as <a href="/wiki/Server-side_include" class="mw-redirect" title="Server-side include">server-side includes</a> or <a href="/wiki/Common_Gateway_Interface" title="Common Gateway Interface">CGI</a> support. These features were not common on early web servers accessible to the public.\n</p><p>Early websites often used a frame at the top to display a banner which could not be scrolled away. These banner frames sometimes included the site\'s logo as well as advertising.<sup id="cite_ref-4" class="reference"><a href="#cite_note-4">&#91;4&#93;</a></sup>\n</p><p><a href="/wiki/XHTML" title="XHTML">XHTML</a>, intended as a successor to HTML 4, removed all frames in 1.1. The intended eventual replacement was <a href="/wiki/XFrames" title="XFrames">XFrames</a>,<sup id="cite_ref-5" class="reference"><a href="#cite_note-5">&#91;5&#93;</a></sup> which attempts to solve the problem of addressing a populated frameset through composite <a href="/wiki/Uniform_Resource_Identifier" title="Uniform Resource Identifier">URIs</a>.\n</p><p>The later <a href="/wiki/HTML5" title="HTML5">HTML5</a> standard, which took a different approach from HTML 4, also removes framesets.<sup id="cite_ref-6" class="reference"><a href="#cite_note-6">&#91;6&#93;</a></sup> The <code>iframe</code> element, however, remains, with a number of "sandboxing" options intended for sharing content between sites.<sup id="cite_ref-7" class="reference"><a href="#cite_note-7">&#91;7&#93;</a></sup>\n</p>\n<h2><span class="mw-headline" id="Advantages">Advantages</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Frame_(World_Wide_Web)&amp;action=edit&amp;section=3" title="Edit section: Advantages">edit</a><span class="mw-editsection-bracket">]</span></span></h2>\n<p>By allowing content to be                                                                                                   loaded and navigated independently, frames offered several advantages over the plain HTML in use when they were first developed:\n</p>\n<ul><li>Simplifying maintenance of content shared across all or most pages, such as navigation data.<sup id="cite_ref-8" class="reference"><a href="#cite_note-8">&#91;8&#93;</a></sup> If an item needs to be added to a sidebar navigation menu, the web page author needs to change only one web page file, whereas each individual page on a traditional non-frameset website would have to be edited if the sidebar menu appeared on all of them.</li>\n<li>Reducing the amount of bandwidth needed by not re-downloading parts of the page which had not changed.</li>\n<li>Allowing several pieces of information to be viewed side by side, with the ability for each section to be scrolled independently. This might include the side-by-side comparison of two pictures or videos, or two different ways to understand something, such as an independently scrolling page of text next to video, images, animation, 3D rotating objects, etc.</li>\n<li>Allowing footnotes or digressions to appear in a dedicated section of the page when linked to, so that the reader does not lose their place in the main text.</li></ul>\n<h2><span class="mw-headline" id="Criticism">Criticism</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Frame_(World_Wide_Web)&amp;action=edit&amp;section=4" title="Edit section: Criticism">edit</a><span class="mw-editsection-bracket">]</span></span></h2>\n<p>The practice of framing HTML content led to numerous criticisms, most centering on <a href="/wiki/Usability" title="Usability">usability</a> and <a href="/wiki/Accessibility" title="Accessibility">accessibility</a> concerns. These include:\n</p>\n<ul><li>Framing breaks the identity between the content and URL as displayed in the browser, making it difficult to link to or bookmark a particular item of content within the frameset<sup id="cite_ref-Nielsen1996_9-0" class="reference"><a href="#cite_note-Nielsen1996-9">&#91;9&#93;</a></sup></li>\n<li>The implementation of frames is inconsistent across different browsers<sup id="cite_ref-Nielsen1996_9-1" class="reference"><a href="#cite_note-Nielsen1996-9">&#91;9&#93;</a></sup></li>\n<li>Browsers which render material linearly do not handle frames well.<sup id="cite_ref-10" class="reference"><a href="#cite_note-10">&#91;10&#93;</a></sup>\n<ul><li><a href="/wiki/Screen_reader" title="Screen reader">Screen reader</a> programs</li>\n<li>text or audio browsers</li>\n<li><a href="/wiki/Email" title="Email">Email</a> browsers such as <a href="/wiki/Agora_(web_browser)" title="Agora (web browser)">Agora</a><sup id="cite_ref-geboyd_11-0" class="reference"><a href="#cite_note-geboyd-11">&#91;11&#93;</a></sup></li>\n<li><a href="/wiki/Mobile_browser" title="Mobile browser">Mobile browsers</a></li></ul></li>\n<li>Framing complicates <a href="/wiki/Web_indexing" title="Web indexing">web indexing</a> and can be detrimental to <a href="/wiki/Search_engine_optimization" title="Search engine optimization">search engine optimization</a>.<sup id="cite_ref-12" class="reference"><a href="#cite_note-12">&#91;12&#93;</a></sup></li>\n<li>Framing confuses the boundaries between content on different servers,<sup id="cite_ref-13" class="reference"><a href="#cite_note-13">&#91;13&#93;</a></sup> which raises issues of copyright infringement<sup id="cite_ref-htmlfaq_14-0" class="reference"><a href="#cite_note-htmlfaq-14">&#91;14&#93;</a></sup></li>\n<li>Visitors arriving from search engines may land on a page intended for display in a frame, resulting in the visitor having no way to navigate to the rest of the site<sup id="cite_ref-prosandcons_15-0" class="reference"><a href="#cite_note-prosandcons-15">&#91;15&#93;</a></sup></li>\n<li>Frames change the behavior of the back button.<sup id="cite_ref-prosandcons_15-1" class="reference"><a href="#cite_note-prosandcons-15">&#91;15&#93;</a></sup></li>\n<li>Users usually do not expect browsers to print frames the way they do.<sup id="cite_ref-htmlfaq_14-1" class="reference"><a href="#cite_note-htmlfaq-14">&#91;14&#93;</a></sup></li>\n<li>External links on web pages which use frames may cause other pages to appear in the frameset, since the default behaviour for a link is to load in the current frame if the author does not specify otherwise. This could be used by unscrupulous webmasters to make it appear as though content from another site was actually part of the site hosting the frameset.</li>\n<li>If the screen resolution or browser window size is too low then each frame will have scroll bars which can look messy and uses up already limited space. Such behaviour typically resulted more from bad site design (fixed layouts instead of fluid layouts), whereby not all frameset features were put into proper use.<br />This behaviour could be mitigated by:\n<ul><li>disabling scrolling for smaller frames that typically did not require a scrollbar;</li>\n<li>using fluid design characteristics in target pages instead of fixed designs, so that the content would not cause horizontal scrollbars in the first place.</li></ul></li></ul>\n<h2><span class="mw-headline" id="Alternatives">Alternatives</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Frame_(World_Wide_Web)&amp;action=edit&amp;section=5" title="Edit section: Alternatives">edit</a><span class="mw-editsection-bracket">]</span></span></h2>\n<p>As web technology developed, many of the purposes for which frames were used became possible in ways that avoided the problems identified with frames.\n</p>\n<ul><li><a href="/wiki/Cascading_Style_Sheets" class="mw-redirect" title="Cascading Style Sheets">Cascading Style Sheets</a> (CSS) allowed elements of a page to be scrolled independently (using the <code>overflow</code> property) or held on screen while other content is scrolled (using <code>position:fixed</code>)</li>\n<li><a href="/wiki/Server-Side_Includes" class="mw-redirect" title="Server-Side Includes">Server-Side Includes</a> allowed shared content to be edited once and automatically delivered to the client as part of a finished page; as server CPU and connection speeds increased, the extra work required to do this on the fly became a lesser consideration.</li>\n<li><a href="/wiki/Common_Gateway_Interface" title="Common Gateway Interface">CGI</a> and web-oriented <a href="/wiki/Scripting_languages" class="mw-redirect" title="Scripting languages">scripting languages</a> and <a href="/wiki/Web_development" title="Web development">web development</a> frameworks such as <a href="/wiki/PHP" title="PHP">PHP</a> and <a href="/wiki/Active_Server_Pages" title="Active Server Pages">Active Server Pages</a>, as well as database-backed <a href="/wiki/Content_Management_System" class="mw-redirect" title="Content Management System">Content Management Systems</a> such as <a href="/wiki/WordPress" title="WordPress">WordPress</a>, provided much richer options for maintaining content and providing navigation.</li>\n<li><a href="/wiki/Client-side_scripting" class="mw-redirect" title="Client-side scripting">Client-side scripting</a> and <a href="/wiki/Dynamic_HTML" title="Dynamic HTML">Dynamic HTML</a> allowed parts of a page to be visually replaced based on a user\'s actions. This allowed much more flexibility for showing "side" content, such as footnotes or instructions, as these could now be displayed and hidden anywhere on the page rather than requiring a pre-defined frame.</li>\n<li><a href="/wiki/AJAX" class="mw-redirect" title="AJAX">AJAX</a> allowed for dynamic display within a page of content even when it needs to be fetched from the server, for instance based on the logged in user or events elsewhere.</li></ul>\n<p>Not all of problems identified with framesets are eliminated by using these alternative approaches; for instance issues with Back/Forward navigation, bookmarking, and indexing remain on many sites which make heavy use of DHTML / AJAX navigation.\n</p>\n<h2><span class="mw-headline" id="See_also">See also</span><span
