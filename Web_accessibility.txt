accessibility</b> is the <a href="/wiki/Inclusion_(disability_rights)" title="Inclusion (disability rights)">inclusive practice</a> of ensuring there are no barriers that prevent interaction with, or access to, <a href="/wiki/Website" title="Website">websites</a> on the <a href="/wiki/World_Wide_Web" title="World Wide Web">World Wide Web</a> by people with physical <a href="/wiki/Disabilities" class="mw-redirect" title="Disabilities">disabilities</a>, situational disabilities, and socio-economic restrictions on bandwidth and speed. When sites are correctly designed, developed and edited, generally all users have equal <a href="/wiki/Access_to_information" class="mw-redirect" title="Access to information">access to information</a> and functionality.\n</p><p>For example, when a site is coded with semantically meaningful <a href="/wiki/HTML" title="HTML">HTML</a>, with textual equivalents provided for images and with links named meaningfully, this helps blind users using text-to-speech software and/or text-to-Braille hardware. When text and images are large and/or enlargeable, it is easier for users with poor sight to read and understand the content. When links are underlined (or otherwise differentiated) as well as colored, this ensures that <a href="/wiki/Color_blind" class="mw-redirect" title="Color blind">color blind</a> users will be able to notice them. When <a href="/wiki/Clickable" class="mw-redirect" title="Clickable">clickable</a> links and areas are large, this helps users who cannot control a <a href="/wiki/Computer_mouse" title="Computer mouse">mouse</a> with precision. When pages are not coded in a way that hinders navigation by means of the <a href="/wiki/Computer_keyboard" class="mw-redirect" title="Computer keyboard">keyboard</a> alone, or a single <a href="/wiki/Switch_access" title="Switch access">switch access</a> device alone, this helps users who cannot use a mouse or even a standard keyboard. When videos are <a href="/wiki/Closed_caption" class="mw-redirect" title="Closed caption">closed captioned</a> or a <a href="/wiki/Sign_language" title="Sign language">sign language</a> version is available, deaf and hard-of-hearing users can understand the video. When flashing effects are avoided or made optional, users prone to <a href="/wiki/Seizure" title="Seizure">seizures</a> caused by these effects are not put at risk. And when content is written in plain language and illustrated with instructional diagrams and animations, users with <a href="/wiki/Dyslexia" title="Dyslexia">dyslexia</a> and <a href="/wiki/Learning_disabilities" class="mw-redirect" title="Learning disabilities">learning difficulties</a> are better able to understand the content. When sites are correctly built and maintained, all of these users can be accommodated without decreasing the <a href="/wiki/Usability" title="Usability">usability</a> of the site for non-disabled users.\n</p><p>The needs that Web accessibility aims to address include:\n</p>\n<ul><li><b>Visual:</b> <a href="/wiki/Visual_impairment" title="Visual impairment">Visual impairments</a> including <a href="/wiki/Blindness" class="mw-redirect" title="Blindness">blindness</a>, various common types of <a href="/wiki/Low_vision" class="mw-redirect" title="Low vision">low vision</a> and poor eyesight, various types of <a href="/wiki/Color_blindness" title="Color blindness">color blindness</a>;</li>\n<li><b>Motor/mobility:</b> e.g. difficulty or inability to use the hands, including tremors, muscle slowness, loss of fine muscle control, etc., due to conditions such as <a href="/wiki/Parkinson%27s_disease" title="Parkinson&#39;s disease">Parkinson\'s disease</a>, <a href="/wiki/Muscular_dystrophy" title="Muscular dystrophy">muscular dystrophy</a>, <a href="/wiki/Cerebral_palsy" title="Cerebral palsy">cerebral palsy</a>, <a href="/wiki/Stroke" title="Stroke">stroke</a>;</li>\n<li><b>Auditory:</b> <a href="/wiki/Deafness" title="Deafness">Deafness</a> or <a href="/wiki/Hearing_impairment" class="mw-redirect" title="Hearing impairment">hearing impairments</a>, including individuals who are hard of hearing;</li>\n<li><b>Seizures:</b> Photo epileptic <a href="/wiki/Seizure" title="Seizure">seizures</a> caused by visual strobe or flashing effects.</li>\n<li><b>Cognitive and intellectual:</b> <a href="/wiki/Developmental_disabilities" class="mw-redirect" title="Developmental disabilities">Developmental disabilities</a>, <a href="/wiki/Learning_disability" title="Learning disability">learning difficulties</a> (<a href="/wiki/Dyslexia" title="Dyslexia">dyslexia</a>, <a href="/wiki/Dyscalculia" title="Dyscalculia">dyscalculia</a>, etc.), and <a href="/wiki/Cognitive_disabilities" class="mw-redirect" title="Cognitive disabilities">cognitive disabilities</a> (PTSD, Alzheimer\'s) of various origins, affecting memory, attention, developmental "maturity", problem-solving and logic skills, etc.</li></ul>\n<p>Accessibility is not confined to the list above, rather it extends to anyone who is experiencing any permanent, temporary or situational disability. Situational disability refers to someone who may be experiencing a boundary based on the current experience. For example, a person may be situationally one-handed if they are carrying a baby. Web accessibility should be mindful of users experiencing a wide variety of barriers. Unfortunately, according to a 2018 WebAIM global survey of web accessibility practitioners, close to 93% of survey respondents received no formal schooling on web accessibility.<sup id="cite_ref-1" id="Assistive_technologies_used_for_web_browsing">Assistive technologies used for web browsing</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Web_accessibility&amp;action=edit&amp;section=1" title="Edit section: Assistive technologies used for web browsing">edit</a><span class="mw-editsection-bracket">]</span></span></h2>\n<p>Individuals living with a disability use <a href="/wiki/Assistive_technology" title="Assistive technology">assistive technologies</a> such as the following to enable and assist web browsing:\n</p>\n<ul><li><a href="/wiki/Screen_reader" title="Screen reader">Screen reader</a> software, which can read out, using synthesized speech, either selected elements of what is being displayed on the monitor (helpful for users with reading or learning difficulties), or which can read out everything that is happening on the computer (used by blind and vision impaired users).</li>\n<li>Braille terminals, consisting of a <a href="/wiki/Refreshable_braille_display" title="Refreshable braille display">refreshable braille display</a> which renders text as <a href="/wiki/Braille" title="Braille">braille</a> characters (usually by means of raising pegs through holes in a flat surface) and either a mainstream keyboard or a braille keyboard.</li>\n<li><a href="/wiki/Screen_magnifier" title="Screen magnifier">Screen magnification</a> software, which enlarges what is displayed on the computer monitor, making it easier to read for vision impaired users.</li>\n<li><a href="/wiki/Speech_recognition" title="Speech recognition">Speech recognition</a> software that can accept spoken commands to the computer, or turn dictation into grammatically correct text \xe2\x80\x93 useful for those who have difficulty using a mouse or a keyboard.</li>\n<li>Keyboard overlays, which can make typing easier or more accurate for those who have motor control difficulties.</li>\n<li>Access to subtitled or sign language videos for deaf people.</li></ul>\n<h2><span class="mw-headline" id="Guidelines_on_accessible_web_design">Guidelines on accessible web design</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Web_accessibility&amp;action=edit&amp;section=2" title="Edit section: Guidelines on accessible web design">edit</a><span class="mw-editsection-bracket">]</span></span></h2>\n<h3><span class="mw-headline" id="Web_Content_Accessibility_Guidelines">Web Content Accessibility Guidelines</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Web_accessibility&amp;action=edit&amp;section=3" title="Edit Recommendation. WCAG 2.0 aims to be up to date and more technology neutral. Though web designers can choose either standard to follow, the WCAG 2.0 have been widely accepted as the definitive guidelines on how to create accessible websites. Governments are steadily adopting the WCAG 2.0 as the accessibility standard for their own websites.<sup id="cite_ref-2" class="reference"><a href="#cite_note-2">&#91;2&#93;</a></sup> In 2012, the Web id="cite_ref-3" class="reference"><a href="#cite_note-3">&#91;3&#93;</a></sup>\n</p>\n<h4><span class="mw-headline" id="Criticism_of_WAI_guidelines">Criticism of WAI guidelines</span><span the process.<sup id="cite_ref-4" class="reference"><a href="#cite_note-4">&#91;4&#93;</a></sup> There was a formal objection to WCAG\'s original claim that WCAG 2.0 will address requirements for people with learning disabilities and cognitive limitations headed by <a href="/wiki/Lisa_Seeman" title="Lisa Seeman">Lisa Seeman</a> and signed by 40 organizations and people.<sup id="cite_ref-5" class="reference"><a href="#cite_note-5">&#91;5&#93;</a></sup> In articles such as "WCAG 2.0: The new W3C guidelines evaluated",<sup id="cite_ref-6" class="reference"><a href="#cite_note-6">&#91;6&#93;</a></sup> "To Hell with WCAG 2.0"<sup id="cite_ref-7" class="reference"><a href="#cite_note-7">&#91;7&#93;</a></sup> and "Testability Costs Too Much",<sup id="cite_ref-8" class="reference"><a href="#cite_note-8">&#91;8&#93;</a></sup> the WAI has been criticised for allowing WCAG 1.0 to get increasingly out of step with today\'s technologies and techniques for creating and consuming web content, for the slow pace of development of WCAG 2.0, for making the new guidelines difficult to navigate and understand, and other argued failings.\n</p>\n<h2><span class="mw-headline" id="Essential_components_of_web_accessibility">Essential components of web accessibility</span><span class="mw-editsection"><span create web <b>content</b>. <br /><b>People</b> ("<b>users</b>") use web <b>browsers</b>, <b>media players</b>, <b>assistive technologies</b> or other "<b>user agents</b>" to get and interact with the <b>content</b>.<sup id="cite_ref-waicomponents_9-1" class="reference"><a href="#cite_note-waicomponents-9">&#91;9&#93;</a></sup></p></blockquote>\n<h2><span class="mw-headline" class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Web_accessibility&amp;action=edit&amp;section=8" title="Edit section: Web Content Accessibility One approach is to protect access to websites for people with disabilities by using existing human or civil rights legislation. Some countries, like the U.S., protect access for people with disabilities through the technology procurement process.<sup id="cite_ref-16" class="reference"><a href="#cite_note-16">&#91;16&#93;</a></sup> It is common for nations to support and adopt the Web Content Accessibility Guidelines (WCAG) 2.0 by referring to the guidelines in their legislation.<sup id="cite_ref-17" class="reference"><a href="#cite_note-17">&#91;17&#93;</a></sup><sup id="cite_ref-18" class="reference"><a href="#cite_note-18">&#91;18&#93;</a></sup> Compliance with web accessibility guidelines is a legal requirement primarily in North America, Europe, parts of South America and parts of Asia.<sup id="cite_ref-19" class="reference"><a href="#cite_note-19">&#91;19&#93;</a></sup>\n</p>\n<h3><span class="mw-headline" id="Australia">Australia</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Web_accessibility&amp;action=edit&amp;section=11" title="Edit section: Australia">edit</a><span class="mw-editsection-bracket">]</span></span></h3>\n<p>In 2000, an <a href="/wiki/Australian" class="mw-redirect" title="Australian">Australian</a> blind man won a $20,000 court case against the <a href="/wiki/Sydney_Organising_Committee_for_the_Olympic_Games" title="Sydney Organising Committee for the Olympic Games">Sydney Organizing Committee of the Olympic Games (SOCOG)</a>. This was the first successful case under <a href="/wiki/Disability_Discrimination_Act_1992" title="Disability Discrimination Act 1992">Disability Discrimination Act 1992</a> because SOCOG had failed to make their official website, <a href="/wiki/Sydney_Olympic_Games" class="mw-redirect" title="Sydney Olympic Games">Sydney Olympic Games</a>, adequately accessible to blind users. <a href="/w/index.php?title=The_Human_Rights_and_Equal_Opportunity_Commission_(HREOC)&amp;action=edit&amp;redlink=1" class="new" title="The Human Rights and Equal Opportunity Commission (HREOC) (page does not exist)">The Human Rights and Equal Opportunity Commission (HREOC)</a> also published World Wide Web Access: Disability Discrimination Act Advisory Notes.<sup id="cite_ref-20" class="reference"><a href="#cite_note-20">&#91;20&#93;</a></sup> All Governments in Australia also have policies and guidelines that require accessible public websites.\n</p>\n<h3><span class="mw-headline" id="Brazil">Brazil</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Web_accessibility&amp;action=edit&amp;section=12" title="Edit section: Brazil">edit</a><span class="mw-editsection-bracket">]</span></span></h3>\n<p>In <a href="/wiki/Brazil" title="Brazil">Brazil</a>, the federal government published a paper with guidelines for accessibility on 18 January 2005, for public reviewing. On 14 December of the same year, the second version was published, including suggestions made to the first version of the paper. On 7 May 2007, the accessibility guidelines of the paper became compulsory to all federal websites. The current version of the paper, which follows the WCAG 2.0 guidelines, is named <b>e-MAG, Modelo de Acessibilidade de Governo Eletr\xc3\xb4nico</b> (Electronic Government Accessibility Model), and is maintained by Brazilian <a href="/wiki/Ministry_of_Planning,_Budget,_and_Management" class="mw-redirect" title="Ministry of Planning, Budget, and Management">Ministry of Planning, Budget, and Management</a>.\n</p><p>The paper can be viewed and downloaded at its official website.<sup id="cite_ref-21" class="reference"><a href="#cite_note-21">&#91;21&#93;</a></sup>\n</p>\n<h3><span class="mw-headline" id="Canada">Canada</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Web_accessibility&amp;action=edit&amp;section=13" title="Edit section: Canada">edit</a><span class="mw-editsection-bracket">]</span></span></h3>\n<p>In 2011, the Government of Canada began phasing in the implementation of a new set of web standards that are aimed at ensuring government websites are accessible, usable, interoperable and optimized for mobile devices. These standards replace Common Look and Feel 2.0 (CLF 2.0) Standards for the Internet.\n</p><p>The first of these four standards, Standard on Web Accessibility<sup id="cite_ref-22" class="reference"><a href="#cite_note-22">&#91;22&#93;</a></sup> came into full effect on 31 July 2013. The Standard on Web Accessibility follows the <a href="/wiki/Web_Content_Accessibility_Guidelines" title="Web Content Accessibility Guidelines">Web Content Accessibility Guidelines</a> (WCAG) 2.0 AA, and contains a list of exclusions that is updated annually. It is accompanied by an explicit Assessment Methodology<sup id="cite_ref-23" class="reference"><a href="#cite_note-23">&#91;23&#93;</a></sup> that helps government departments comply. The government also developed the Web Experience Toolkit (WET),<sup id="cite_ref-24" class="reference"><a href="#cite_note-24">&#91;24&#93;</a></sup> a set of reusable web components for building innovative websites.  The WET helps government departments build innovative websites that are accessible, usable and interoperable and therefore comply with the government\'s standards. The <a rel="nofollow" class="external text" href="https://wet-boew.github.io/wet-boew/index.html">WET toolkit is open source</a> and available for anyone to use.\n</p><p>The three related web standards are:  the Standard on Optimizing Websites and Applications for Mobile Devices,<sup id="cite_ref-25" class="reference"><a href="#cite_note-25">&#91;25&#93;</a></sup> the Standard on Web Usability<sup id="cite_ref-26" class="reference"><a href="#cite_note-26">&#91;26&#93;</a></sup> and the Standard on Web Interoperability.<sup id="cite_ref-27" class="reference"><a href="#cite_note-27">&#91;27&#93;</a></sup>\n</p><p>In 2019 the <a href="/wiki/Government_of_Canada" title="Government of Canada">Government of Canada</a> passed the <a href="/wiki/Accessible_Canada_Act" title="Accessible Canada Act">Accessible Canada Act</a>. This builds on the on provincial legislation like the <a href="/wiki/Accessibility_for_Ontarians_with_Disabilities_Act,_2005" title="Accessibility for Ontarians with Disabilities Act, 2005">Accessibility for Ontarians with Disabilities Act</a>, <a href="/wiki/The_Accessibility_for_Manitobans_Act" title="The Accessibility for Manitobans Act">The Accessibility for Manitobans Act</a> and the <a href="/wiki/Nova_Scotia_Accessibility_Act" title="Nova Scotia Accessibility Act">Nova Scotia Accessibility Act</a>.\n</p>\n<h3><span class="mw-headline" id="European_Union">European Union</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Web_accessibility&amp;action=edit&amp;section=14" title="Edit section: European Union">edit</a><span class="mw-editsection-bracket">]</span></span></h3>\n<p>In February 2014 a draft law was endorsed by the <a href="/wiki/European_Parliament" title="European Parliament">European Parliament</a> stating that all websites managed by public sector bodies have to be made accessible to everyone.<sup id="cite_ref-28" class="reference"><a href="#cite_note-28">&#91;28&#93;</a></sup>\n</p><p>On 26 October 2016, the European Parliament approved the <a href="/wiki/Web_Accessibility_Directive" title="Web Accessibility Directive">Web Accessibility Directive</a> that requires that the websites and mobile apps of public sector bodies be accessible. The relevant accessibility requirements are described in the European standard <a href="/wiki/EN_301_549" title="EN 301 549">EN 301 549</a> V1.1.2 (published by <a href="/wiki/ETSI" title="ETSI">ETSI</a>). PAS 78, produced by the <a href="/wiki/Disability_Rights_Commission" title="Disability Rights Commission">Disability Rights Commission</a> and usable by disabled people. Both members of the public and federal employees have the right to access this technology, such as computer hardware and software, websites, phone systems, and copiers.<sup id="cite_ref-51" class="reference"><a href="#cite_note-51">&#91;51&#93;</a></sup>  \nAlso, <a href="/wiki/Section_504_of_the_Rehabilitation_Act" title="Section 504 of the Rehabilitation Act">Section 504 of the Rehabilitation Act</a> prohibits discrimination on the basis of disability for entities receiving federal funds and has been cited in multiple lawsuits against organizations such as hospitals that receive federal funds through medicare/medicaid.\n</p><p>In addition, <a href="/wiki/Americans_with_Disabilities_Act_of_1990#Title_III\xe2\x80\x94public_accommodations_(and_commercial_facilities)" title="Americans with Disabilities Act of 1990">Title III of the Americans with Disabilities Act (ADA)</a> prohibits discrimination on the basis of disability.  There is some debate on the matter; multiple courts and the U.S. Department of Justice have taken the position that the ADA requires website and app operators and owners to take affirmative steps to make their websites and apps accessible to disabled persons and compatible with common assistive technologies such as the JAWS screen reader, while other courts have taken the position that the ADA does not apply online.  The U.S. Department of Justice has endorsed the WCAG2.0AA standard as an appropriate standard for accessibility in multiple settlement agreements.<sup id="cite_ref-52" class="reference"><a href="#cite_note-52">&#91;52&#93;</a></sup>\n</p><p>Numerous lawsuits challenging websites and mobile apps on the basis of the ADA have been filed since 2017. These cases appears spurred by a 2017 case, <i>Gil v. Winn Dixie Stores</i>, in which a federal court in Florida ruled that <a href="/wiki/Winn_Dixie" class="mw-redirect" title="Winn Dixie">Winn Dixie</a>\'s website must be accessible. Around 800 cases related to web accessibility were filed in 2017, and over 2,200 were filed in 2018. Additionally, though the Justice Department had stated in 2010 that they would publish guidelines for web accessibility, they reversed this plan in 2017, also spurring legal action against inaccessible sites.<sup id="cite_ref-dominos_53-0" class="reference"><a href="#cite_note-dominos-53">&#91;53&#93;</a></sup>\n</p><p>A notable lawsuit related to the ADA was filed against <a href="/wiki/Domino%27s_Pizza" title="Domino&#39;s Pizza">Domino\'s Pizza</a> by a blind user who could not use Domino\'s mobile app. At the federal district level, the court rules in favor of Domino\'s as the Justice Department had not established the guidelines for accessibility, but this was appealed to the <a href="/wiki/Ninth_Circuit" class="mw-redirect" title="Ninth Circuit">Ninth Circuit</a>. The Ninth Circuit overruled the district court, ruling that because Domino\'s is a brick-and-mortar store, which must meet the ADA, and the mobile app an extension of their services, their app must also be compliant with the ADA. Domino\'s has petitioned to the Supreme Court, backed by many other restaurants and retail chains, arguing that this decision impacts their <a href="/wiki/Due_Process_Clause" title="Due Process Clause">Due Process</a> since handicapped customers have other, more accessible means to order.<sup id="cite_ref-dominos_53-1" class="reference"><a href="#cite_note-dominos-53">&#91;53&#93;</a></sup> In October 2019, the Supreme Court declined to hear the case, which effectively upheld the decision of the 9th Circuit Court and requires the case to be heard as it stands.<sup id="cite_ref-54" class="reference"><a href="#cite_note-54">&#91;54&#93;</a></sup>\n</p><p>The number and cost of federal accessibility lawsuits has risen dramatically in the last few years.<sup id="cite_ref-55" class="reference"><a href="#cite_note-55">&#91;55&#93;</a></sup>\n</p>\n<h2><span class="mw-headline" id="Website_accessibility_audits">Website accessibility audits</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Web_accessibility&amp;action=edit&amp;section=26" title="Edit section: Website accessibility audits">edit</a><span class="mw-editsection-bracket">]</span></span></h2>\n<p>A growing number of organizations, companies and consultants offer <i>website accessibility audits</i>. These audits, a type of <a href="/wiki/System_testing" title="System testing">system testing</a>, identify accessibility problems that exist within a website, and provide advice and guidance on the steps that need to be taken to correct these problems.\n</p><p>A range of methods are used to audit websites for <a href="/wiki/Accessibility" title="Accessibility">accessibility</a>:\n</p>\n<ul><li>Automated tools are available which can identify some of the problems that are present. Depending on the tool the result may vary widely making it difficult to compare test results.<sup id="cite_ref-56" class="reference"><a href="#cite_note-56">&#91;56&#93;</a></sup></li>\n<li>Expert technical reviewers, knowledgeable in web design technologies and accessibility, can review a representative selection of pages and provide detailed feedback and advice based on their findings.</li>\n<li>User testing, usually overseen by technical experts, involves setting tasks for ordinary users to carry out on the website, and reviewing the problems these users encounter as they try to carry out the tasks.</li></ul>\n<p>Each of these methods has its strengths and weaknesses:\n</p>\n<ul><li>Automated tools can process many pages in a relatively short length of time, but can only identify some of the accessibility problems that might be present in the website.</li>\n<li>Technical expert review will identify many of the problems that exist, but the process is time consuming, and many websites are too large to make it possible for a person to review every page.</li>\n<li>User testing combines elements of usability and accessibility testing, and is valuable for identifying problems that might otherwise be overlooked, but needs to be used knowledgeably to avoid the risk of basing design decisions on one user\'s preferences.</li></ul>\n<p>Ideally, a combination of methods should be used to assess the accessibility of a website.\n</p>\n<h2><span class="mw-headline" id="Remediating_inaccessible_websites">Remediating inaccessible websites</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Web_accessibility&amp;action=edit&amp;section=27" title="Edit section: Remediating inaccessible websites">edit</a><span class="mw-editsection-bracket">]</span></span></h2>\n<p>Once an accessibility audit has been conducted, and accessibility errors have been identified, the errors will need to be remediated in order to ensure the site is compliant with accessibility errors. The traditional way of correcting an inaccessible site is to go back into the source code, reprogram the error, and then test to make sure the bug was fixed. If the website is not scheduled to be revised in the near future, that error (and others) would remain on the site for a lengthy period of time, possibly violating accessibility guidelines. Because this is a complicated process, many website owners choose to build accessibility into a new site design or re-launch, as it can be more efficient to develop the site to comply with accessibility guidelines, rather than to remediate errors later.\n</p><p>With the progress in <a href="/wiki/AI" class="mw-redirect" title="AI">AI</a> technology, web accessibility has become more accessible. With 3rd party addons that leverage AI and <a href="/wiki/Machine_learning" title="Machine learning">machine learning</a>,  it is possible to offer changes to the <a href="/wiki/Website_design" class="mw-redirect" title="Website design">website design</a> without altering the source code. This way, a website can be accessible to different types of users without the need to adjust the website for every accessibility equipment.<sup id="cite_ref-57" class="reference"><a href="#cite_note-57">&#91;57&#93;</a></sup>\n</p>\n<h2><span class="mw-headline" id="Accessible_Web_applications_and_WAI-ARIA">Accessible Web applications and WAI-ARIA</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Web_accessibility&amp;action=edit&amp;section=28" title="Edit section: Accessible Web applications and WAI-ARIA">edit</a><span class="mw-editsection-bracket">]</span></span></h2>\n<p>For a web page to be accessible all important semantics about the page\'s functionality must be available so that assistive technology can understand and process the content and adapt it for the user.  \nHowever, as content becomes more and more complex, the standard HTML tags and attributes become inadequate in providing semantics reliably. Modern Web applications often apply scripts to elements to control their functionality and to enable them to act as a control or other dynamic component. These custom components or widgets do not provide a way to convey semantic information to the user agent. <a href="/wiki/WAI-ARIA" title="WAI-ARIA">WAI-ARIA</a> (Accessible Rich Internet Applications) is a specification<sup id="cite_ref-58" class="reference"><a href="#cite_note-58">&#91;58&#93;</a></sup> published by the <a href="/wiki/World_Wide_Web_Consortium" title="World Wide Web Consortium">World Wide Web Consortium</a> that specifies how to increase the <a href="/wiki/Accessibility" title="Accessibility">accessibility</a> of <a href="/wiki/Dynamic_content" class="mw-redirect" title="Dynamic content">dynamic content</a> and <a href="/wiki/Elements_of_graphical_user_interfaces" class="mw-redirect" title="Elements of graphical user interfaces">user interface components</a> developed with <a href="/wiki/Ajax_(programming)" title="Ajax (programming)">Ajax</a>, <a href="/wiki/HTML" title="HTML">HTML</a>, <a href="/wiki/JavaScript" title="JavaScript">JavaScript</a> and related technologies. ARIA enables accessibility by enabling the author to provide all the semantics to fully describe its supported behaviour. It also allows each element to expose its current states and properties and its relationships between other elements. Accessibility problems with the focus and tab index are also corrected.\n</p>\n<h2><span class="mw-headline" id="See_also">See also</span><span class="mw-editsection"><span
