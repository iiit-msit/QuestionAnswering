1.1)">1.1</a></li></ul></li></ul></td>\n</tr></tbody></table>\n<p>Web pages authored using <b>hypertext markup language</b> (<a href="/wiki/HTML_email" title="HTML email">HTML</a>) may contain multilingual text represented with the <b>Unicode universal character set</b>. Key to the relationship between Unicode and HTML is the relationship between the "document character set", which defines the set of characters that may be present in a HTML document and assigns numbers to them, and the "external character encoding", or "charset", used to encode a given document as a sequence of bytes.\n</p><p>In RFC 1866, the initial HTML 2.0 standard, the document character set was defined as ISO-8859-1. It was extended to <a href="/wiki/ISO_10646" class="mw-redirect" title="ISO 10646">ISO 10646</a> (which is basically equivalent to Unicode) by RFC 2070. It does not vary between documents of different languages or created on different platforms. The external character encoding is chosen by the author of the document (or the software the author uses to create the document) and determines how the bytes used to store and/or transmit the document map to characters from the document character set. Characters not present in the chosen external character encoding may be represented by character entity references.\n</p><p>The relationship between <a href="/wiki/Unicode" title="Unicode">Unicode</a> and HTML tends to be a difficult topic for many computer professionals, document authors, and <a href="/wiki/World_Wide_Web" title="World Wide Web">web</a> users alike. The accurate representation of text in <a href="/wiki/Web_page" title="Web page">web pages</a> from different <a href="/wiki/Natural_language" title="Natural language">natural languages</a> and <a href="/wiki/Writing_system" title="Writing system">writing systems</a> is complicated by the details of <a href="/wiki/Character_encoding" title="Character encoding">character encoding</a>, <a href="/wiki/Markup_language" title="Markup language">markup language</a> syntax, <a href="/wiki/Computer_font" title="Computer font">font</a>, and varying levels of support by <a characters</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Unicode_and_HTML&amp;action=edit&amp;section=1" title="Edit section: HTML document characters">edit</a><span class="mw-editsection-bracket">]</span></span></h2>\n<p>Web pages are typically <a href="/wiki/HTML" title="HTML">HTML</a> or <a href="/wiki/XHTML" title="XHTML">XHTML</a> documents. Both types of documents consist, at a fundamental level, of <a href="/wiki/Character_(computing)" title="Character (computing)">characters</a>, which are <a href="/wiki/Grapheme" title="Grapheme">graphemes</a> and grapheme-like units, independent of how they manifest in <a href="/wiki/Computer_storage" class="mw-redirect" title="Computer storage">computer storage</a> systems and <a href="/wiki/Computer_network" title="Computer network">networks</a>.\n</p><p>An HTML document is a sequence of Unicode characters. More specifically, HTML 4.0 documents are required to consist of characters in the HTML <i>document character set</i>&#160;: a character repertoire wherein each character is assigned a unique, non-negative integer <i>code point</i>. This set is defined in the HTML 4.0 <a href="/wiki/Document_Type_Definition" class="mw-redirect" title="Document Type Definition">DTD</a>, which also establishes the syntax (allowable sequences of characters) that can produce a valid HTML document. The HTML document character set for HTML 4.0 consists of most, but not all, of the characters jointly defined by <a href="/wiki/Unicode" title="Unicode">Unicode</a> and ISO/IEC 10646: the <a href="/wiki/Universal_Character_Set" class="mw-redirect" title="Universal Character Set">Universal Character Set</a> (UCS).\n</p><p>Like HTML documents, an XHTML document is a sequence of Unicode characters. However, an XHTML document is an <a href="/wiki/XML" title="XML">XML</a> document, which, while not having an explicit "document character" layer of <a href="/wiki/Abstraction" title="Abstraction">abstraction</a>, nevertheless relies upon a similar definition of permissible characters that cover most, but not all, of the Unicode/UCS character definitions. The sets used by HTML and XHTML/XML are slightly different, but these differences have little effect on the average document author.\n</p><p>Regardless of whether the document is HTML or XHTML, when stored on a <a href="/wiki/File_system" title="File system">file system</a> or transmitted over a network, the document\'s characters are <i>encoded</i> as a sequence of <a href="/wiki/Bit" title="Bit">bit</a> <a href="/wiki/Octet_(computing)" title="Octet (computing)">octets</a> (<i><a href="/wiki/Byte" title="Byte">bytes</a></i>) according to a particular character encoding. This encoding may either be a <a href="/wiki/Unicode_Transformation_Format" class="mw-redirect" title="Unicode Transformation Format">Unicode Transformation Format</a>, like <a href="/wiki/UTF-8" title="UTF-8">UTF-8</a>, that can directly encode any Unicode character, or a legacy encoding, like <a href="/wiki/Windows-1252" title="Windows-1252">Windows-1252</a>, that cannot. However, even when using encodings that do not support all Unicode characters, the encoded document may make use of <a href="/wiki/Numeric_character_references" class="mw-redirect" title="Numeric character references">numeric character references</a>. For example, <code>&amp;#x263A;</code> (\xe2\x98\xba) is used to indicate a smiling face character in the Unicode character set.\n</p>\n<h3><span class="mw-headline" id="Character_encoding">Character encoding</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Unicode_and_HTML&amp;action=edit&amp;section=2" title="Edit section: Character encoding">edit</a><span class="mw-editsection-bracket">]</span></span></h3>\n<p>In order to support all Unicode characters without resorting to numeric character references, a web page must have an encoding covering all of Unicode. The most popular is <a href="/wiki/UTF-8" title="UTF-8">UTF-8</a>, where the <a href="/wiki/ASCII" title="ASCII">ASCII</a> characters, such as English letters, digits, and some other common characters are preserved unchanged against ASCII. This makes HTML code (such as &lt;br&gt; and &lt;/div&gt;) unchanged compared to ASCII. Characters outside the ASCII range are stored in 2-4 bytes. It is also possible to use <a href="/wiki/UTF-16" title="UTF-16">UTF-16</a> where most characters are stored as two bytes with varying <a href="/wiki/Endianness" title="Endianness">endianness</a>, which is supported by modern browsers but less commonly used.\n</p>\n<h3><span class="mw-headline" id="Numeric_character_references">Numeric character references</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Unicode_and_HTML&amp;action=edit&amp;section=3" title="Edit section: Numeric character references">edit</a><span class="mw-editsection-bracket">]</span></span></h3>\n<div role="note" class="hatnote navigation-not-searchable">Main article: <a href="/wiki/Numeric_character_reference" title="Numeric character reference">Numeric character reference</a></div>\n<p>In order to work around the limitations of legacy encodings, HTML is designed such that it is possible to represent characters from the whole of Unicode inside an HTML document by using a <a href="/wiki/Numeric_character_reference" title="Numeric character reference">numeric character reference</a>: a sequence of characters that explicitly spell out the Unicode code point of the character being represented. A character reference takes the form <b><code>&amp;#</code></b><var>N</var><b><code>;</code></b>, where <var>N</var> is either a <a href="/wiki/Decimal" title="Decimal">decimal</a> number for the Unicode code point, or a <a href="/wiki/Hexadecimal" title="Hexadecimal">hexadecimal</a> number, in which case it must be prefixed by <code>x</code>. The characters that compose the numeric character reference are universally representable in every encoding approved for use on the Internet.\n</p><p>The support for hexadecimal in this context is more recent, so older browsers might have problems displaying characters referenced with hexadecimal numbers\xe2\x80\x94but they will probably have a problem displaying Unicode characters above code point 255 anyway. To ensure better compatibility with older browsers, it is still a common practice to convert the hexadecimal code point into a decimal value (for example <code>&amp;#21512;</code> instead of <code>&amp;#x5408;</code>).\n</p>\n<h3><span class="mw-headline" id="Named_character_entities">Named character entities</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Unicode_and_HTML&amp;action=edit&amp;section=4" title="Edit section: Named character entities">edit</a><span class="mw-editsection-bracket">]</span></span></h3>\n<div role="note" class="hatnote navigation-not-searchable">Main article: <a href="/wiki/Character_entity_reference" class="mw-redirect" title="Character entity reference">character entity reference</a></div>\n<p>In HTML 4, there is a standard set of 252 named <i>character entities</i> for characters - some common, some obscure - that are either not found in certain character encodings or are markup sensitive in some contexts (for example angle brackets and quotation marks). Although any Unicode character can be referenced by its numeric code point, some HTML document authors prefer to use these named entities instead, where possible, as they are less cryptic and were better supported by early browsers.\n</p><p>Character entities can be included in an HTML document via the use of <i>entity references</i>, which take the form <b><code>&amp;</code></b><var>EntityName</var><b><code>;</code></b>, where <var>EntityName</var> is the name of the entity. For example, <code>&amp;mdash;</code>, much like <code>&amp;#8212;</code> or <code>&amp;#x2014;</code>, represents <span title="\xe2\x80\x94"><a href="/wiki/Unicode#Upluslink" title="Unicode">U+</a>2014</span>: the <a href="/wiki/Em_dash" class="mw-redirect" title="Em dash">em dash</a> character "&#8212;" even if the character encoding used doesn\'t contain that character.\n</p><p>For the full list, see: <a href="/wiki/List_of_XML_and_HTML_character_entity_references" title="List of XML and HTML character entity references">List of XML and HTML character entity references</a>.\n</p>\n<h2><span class="mw-headline" id="Character_encoding_determination">Character encoding determination</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Unicode_and_HTML&amp;action=edit&amp;section=5" title="Edit section: Character encoding determination">edit</a><span class="mw-editsection-bracket">]</span></span></h2>\n<p>In order to correctly process HTML, a web browser must ascertain which Unicode characters are represented by the encoded form of an HTML document. In order to do this, the web browser must know what encoding was used.\n</p>\n<h3><span class="mw-headline" id="Encoding_information">Encoding information</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Unicode_and_HTML&amp;action=edit&amp;section=6" title="Edit section: Encoding information">edit</a><span class="mw-editsection-bracket">]</span></span></h3>\n<p>When a document is transmitted via a <a href="/wiki/MIME" title="MIME">MIME</a> message or a transport that uses MIME content types such as an <a href="/wiki/HTTP" class="mw-redirect" title="HTTP">HTTP</a> response, the message may signal the encoding via a Content-Type header, such as <code>Content-Type: text/html; charset=UTF-8</code>. Other external means of declaring encoding are permitted but rarely used. If the document uses a  <a href="/wiki/Comparison_of_Unicode_encodings" title="Comparison of Unicode encodings">Unicode encoding</a>, the encoding info might also be present in the form of a <a href="/wiki/Byte_order_mark" title="Byte order mark">Byte order mark</a>.  Finally, the encoding can be declared via the HTML syntax. For the <code>text/html</code> serialisation then, as long as the page is encoded in an extension of <a href="/wiki/ASCII" title="ASCII">ASCII</a> (such as <a href="/wiki/UTF-8" title="UTF-8">UTF-8</a>, and thus, not if the page is using <a href="/wiki/UTF-16" title="UTF-16">UTF-16</a>), a <code>meta</code> element, like <code>&lt;meta http-equiv="content-type" content="text/html; charset=UTF-8"&gt;</code> or (starting with <a href="/wiki/HTML5" title="HTML5">HTML5</a>) <code>&lt;meta charset="UTF-8"&gt;</code> can be used. For HTML pages serialized as XML, then declaration options is to either rely on the encoding default (which for XML documents is UTF-8), or to use an XML encoding declaration. The meta attribute plays no role in HTML served as XML.\n</p>\n<h3><span class="mw-headline" id="Encoding_defaults">Encoding defaults</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Unicode_and_HTML&amp;action=edit&amp;section=7" title="Edit section: Encoding defaults">edit</a><span class="mw-editsection-bracket">]</span></span></h3>\n<p>An encoding default applies when there is no external or internal encoding declaration and also no Byte order mark. While the encoding default for HTML pages served as XML is required to be UTF-8, the encoding default for a regular Web page (that is: for HTML pages serialized as <code>text/html</code>)  varies depending on the localization of the browser. For a system set up mainly for Western European languages, it will generally be <a href="/wiki/ISO_8859-1#Windows-1252" class="mw-redirect" title="ISO 8859-1">Windows-1252</a>. For Cyrillic alphabet locales, the default is typically <a href="/wiki/Windows-1251" title="Windows-1251">Windows-1251</a>. For a browser from a location where <i>legacy</i> multi-byte character encodings are prevalent, some form of auto-detection is likely to be applied.\n</p>\n<h3><span class="mw-headline" id="Encoding_trends">Encoding trends</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Unicode_and_HTML&amp;action=edit&amp;section=8" title="Edit section: Encoding trends">edit</a><span class="mw-editsection-bracket">]</span></span></h3>\n<p>Because of the legacy of 8-bit text representations in <a href="/wiki/Programming_language" title="Programming language">programming languages</a> and <a href="/wiki/Operating_system" title="Operating system">operating systems</a> and the desire to avoid burdening users with the need to understand the nuances of encoding, many text editors used by HTML authors are unable or unwilling to offer a choice of encodings when saving files to disk and often do not even allow input of characters beyond a very limited range. Consequently, many HTML authors are unaware of encoding issues and may not have any idea what encoding their documents actually use. Misunderstandings, such as the belief that the encoding declaration affects a change in the actual encoding (whereas it is actually just a label that could be inaccurate), is also a reason for this editor attitude. Another factor contributing in the same direction, is the arrival of UTF-8 \xe2\x80\x94 which greatly diminishes the need for other encodings, and thus modern editors tends to default, as recommended by the HTML5 specification,<sup id="cite_ref-1" class="reference"><a href="#cite_note-1">&#91;1&#93;</a></sup> to UTF-8.\n</p>\n<h3><span id="Byte_order_mark.2FUnicode_sniffing"></span><span class="mw-headline" id="Byte_order_mark/Unicode_sniffing">Byte order mark/Unicode sniffing</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Unicode_and_HTML&amp;action=edit&amp;section=9" title="Edit section: Byte order mark/Unicode sniffing">edit</a><span class="mw-editsection-bracket">]</span></span></h3>\n<p>For both serializations of HTML (content-type "text/html" and content/type "application/xhtml+xml"), the Byte order mark (BOM) is an effective way to transmit encoding information within an HTML document. For UTF-8, the BOM is optional, while it is a must for the UTF-16 and the UTF-32 encodings. (Note: UTF-16 and UTF-32 without the BOM are formally known under different names, they are different encodings, and thus needs some form of encoding declaration \xe2\x80\x93 see <a href="/wiki/UTF-16BE" class="mw-redirect" title="UTF-16BE">UTF-16BE</a>, <a href="/wiki/UTF-16LE" class="mw-redirect" title="UTF-16LE">UTF-16LE</a>, <a href="/wiki/UTF-32LE" class="mw-redirect" title="UTF-32LE">UTF-32LE</a> and <a href="/wiki/UTF-32BE" class="mw-redirect" title="UTF-32BE">UTF-32BE</a>.)  The use of the BOM character (U+FEFF) means that the encoding automatically declares itself to any processing application. Processing applications need only look for an initial 0x0000FEFF, 0xFEFF or 0xEFBBBF in the byte stream to identify the document as UTF-32, UTF-16 or UTF-8 encoded respectively. No additional metadata mechanisms are required for these encodings since the byte-order mark includes all of the information necessary for processing applications. In most circumstances the byte-order mark character is handled by editing applications separately from the other characters so there is little risk of an author removing or otherwise changing the byte order mark to indicate the wrong encoding (as can happen when the encoding is declared in English/Latin script). If the document lacks a byte-order mark, the fact that the first non-blank printable character in an HTML document is supposed to be "&lt;" (U+003C) can be used to determine a UTF-8/UTF-16/UTF-32 encoding.\n</p>\n<h3><span class="mw-headline" id="Encoding_overriding">Encoding overriding</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Unicode_and_HTML&amp;action=edit&amp;section=10" title="Edit section: Encoding overriding">edit</a><span class="mw-editsection-bracket">]</span></span></h3>\n<p>Many HTML documents are served with inaccurate encoding information, or no encoding information at all. In order to determine the encoding in such cases, many browsers allow the user to manually select an encoding name from a list. They may also employ an encoding auto-detection algorithm that works in concert <b>with</b> or \xe2\x80\x94 <i>in the case of the BOM and in case of HTML served as XML</i> \xe2\x80\x94 <b>against</b> the manual override.\n</p><p>For HTML documents which are <code>text/html</code> serialized,  manual override may apply to all documents, or only those for which the encoding cannot be ascertained by looking at declarations and/or byte patterns. The fact that the manual override is present and widely used hinders the adoption of accurate encoding declarations on the Web; therefore the problem is likely to persist. But note that Internet Explorer, Chrome and Safari  \xe2\x80\x94 for both XML and <code>text/html</code> serializations \xe2\x80\x94 do not permit the encoding to be overridden whenever the page includes the BOM.<sup id="cite_ref-2" class="reference"><a href="#cite_note-2">&#91;2&#93;</a></sup>\n</p><p>For HTML documents serialized with the preferred XML label \xe2\x80\x94 <code>application/xhtml+xml</code>, manual encoding override is not permitted. To override the encoding of such an XML document would mean that the document stopped being XML, as it is a fatal error for XML documents to have an encoding declaration with detectable errors. Currently,  Gecko browsers such as Firefox, abide to this rule, whereas the bulk of the other common browsers that support HTML as XML, such as Webkit browsers (Chrome/Safari) <sup id="cite_ref-3" class="reference"><a href="#cite_note-3">&#91;3&#93;</a></sup> do allow the encoding of XHTML documents to be manually overridden.\n</p>\n<h2><span class="mw-headline" id="Web_browser_support">Web browser support</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Unicode_and_HTML&amp;action=edit&amp;section=11" title="Edit section: Web browser support">edit</a><span class="mw-editsection-bracket">]</span></span></h2>\n<p>Many browsers are only capable of displaying a small subset of the full Unicode repertoire. title="Mozilla Firefox">Mozilla Firefox</a>, <a href="/wiki/Opera_(web_browser)" title="Opera (web browser)">Opera</a>, <a href="/wiki/Safari_(web_browser)" title="Safari (web browser)">Safari</a> and <a href="/wiki/Internet_Explorer" title="Internet Explorer">Internet Explorer</a> (from version 7 on), are able to display multilingual web pages by intelligently choosing a font to display each individual character on the page. They will correctly display any mix of <a href="/wiki/Unicode_block" title="Unicode block">Unicode blocks</a>, as long as appropriate <a href="/wiki/List_of_typefaces#Unicode_fonts" title="List of typefaces">fonts</a> are present in the <a href="/wiki/Operating_system" title="Operating system">operating system</a>.\n</p><p>Older browsers, such as <a href="/wiki/Netscape_Navigator" title="Netscape Navigator">Netscape Navigator</a> 4.77 and <a href="/wiki/Internet_Explorer_6" title="Internet Explorer 6">Internet Explorer 6</a>, can only display text supported by the current font associated with the character encoding of the page, and may misinterpret numeric character references as being references to code values within the current character encoding, rather than references to Unicode code points. When you are using such a browser, it is unlikely that your computer has all of those fonts, or that the browser can use all available fonts on the same page. As a result, the browser will not display the text in the examples above correctly, though it may display a subset of them. Because they are encoded according to the standard, though, they <i>will</i> display correctly on any system that is compliant and does have the characters available. Further, those characters given names for use in named entity references are likely to be more commonly available than others.\n</p><p>For displaying characters outside the <a href="/wiki/Basic_Multilingual_Plane" class="mw-redirect" title="Basic Multilingual Plane">Basic Multilingual Plane</a>, such as the Gothic letter faihu, which is a variant of the runic letter fehu in
